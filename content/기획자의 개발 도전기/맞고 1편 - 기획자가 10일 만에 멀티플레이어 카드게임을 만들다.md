# 기획자가 10일 만에 멀티플레이어 카드게임을 만들다

## 업무용 앱만 만들다 지쳤다

[[순시미네 4편 - 첫 앱을 앱스토어에 올리기까지|순시미네]]를 만들고, 회사 서비스의 [[1편 - 기획자가 왜 백엔드를 뜯어고치게 되었나|백엔드를 리팩토링]]하면서 바이브 코딩에 꽤 익숙해졌다. 그런데 솔직히 말하면, 요양시설 데이터에 매칭 시스템에 푸시 알림에... 진지한 서비스만 만들다 보니 좀 지쳤다.

그러다 문득 생각했다. **"재미있는 걸 만들어보면 어떨까?"**

우리 회사 캐릭터 '순시미'와 '고냥이'가 있다. 이 캐릭터들로 뭔가 가볍고 재미있는 걸 만들면 브랜드 홍보도 되고, 나도 새로운 도전이 될 것 같았다. 그래서 떠올린 게 **맞고**다.

## 왜 맞고인가

맞고를 선택한 이유는 단순하다.

1. **한국인이라면 누구나 안다** — 룰 설명이 필요 없다 (고 하시겠습니까, 스톱 하시겠습니까)
2. **2인 대전** — 가장 기본적인 멀티플레이어 구조
3. **규칙이 깊다** — 단순해 보이지만 쪽, 뻑, 따닥, 폭탄, 흔들기... 예외 규칙이 끝없다
4. **화투 그래픽이 예쁘다** — 12개월 × 4장 = 48장의 화투 카드 에셋이 화면을 채우면 그럴듯해진다

기획자 입장에서 매력적이었던 건 3번이다. 맞고는 기본 규칙은 간단하지만 **엣지 케이스가 산더미**인 게임이다. 이걸 코드로 옮기는 게 얼마나 어려운지 직접 경험해보고 싶었다.

## Flutter Web을 선택하다

순시미네는 Flutter 네이티브 앱이었다. 이번엔 **Flutter Web**으로 만들기로 했다.

| 비교 | 네이티브 앱 | 웹 앱 |
|------|-----------|-------|
| 설치 | 스토어 다운로드 필요 | URL 클릭만으로 즉시 플레이 |
| 심사 | App Store / Play Store 심사 | 없음 |
| 배포 | 며칠~일주일 | 푸시하면 즉시 반영 |
| 공유 | 앱 링크 전달 | URL 하나 전달 |

카드 게임은 **"야 한판 치자"** 하고 바로 시작할 수 있어야 한다. 앱 설치를 요구하면 이탈률이 높다. URL 하나 공유하면 바로 게임이 시작되는 웹이 맞고에 딱이었다.

물론 Flutter Web이 네이티브 앱만큼의 성능은 안 나온다. 하지만 카드 게임은 FPS 60이 필요한 장르가 아니다. 카드를 놓고, 뒤집고, 가져오는 정도의 애니메이션이면 충분하다.

## 기술 스택

```
프론트: Flutter Web (SDK 3.9)
상태관리: Riverpod
백엔드: Firebase Realtime Database (실시간 대전)
인증: Firebase Auth (Google 로그인 + 익명 로그인)
설정: Firebase Remote Config (디버그 토글)
배포: Vercel
에셋: 화투 카드 48+2장 PNG, Lottie 애니메이션, MP3 효과음
폰트: DoHyeon (도현체, 레트로 감성)
```

순시미네에서는 상태관리를 싱글톤 서비스로 때웠는데, 이번엔 **Riverpod**을 도입했다. 순시미네 이후에 "상태관리 라이브러리라는 게 있더라"는 걸 알았고, 게임처럼 상태 변화가 잦은 앱에서는 제대로 된 상태관리가 필요하다고 판단했다.

Firebase Realtime Database를 선택한 건 **실시간 동기화** 때문이다. 맞고는 내가 패를 내면 상대방 화면에 바로 반영되어야 한다. Supabase Realtime도 있지만, 게임처럼 초 단위로 상태가 바뀌는 경우에는 Firebase Realtime DB가 더 빠르고 안정적이다.

## 10일 스프린트

프로젝트를 12월 10일에 시작해서 12월 19일에 배포했다. **딱 10일.**

순시미네를 만들 때는 몇 달이 걸렸다. 그때와 다른 점이 있다면:

1. **Flutter에 익숙해졌다** — Dart 문법, 위젯 구조, 빌드 사이클을 이미 안다
2. **바이브 코딩 실력이 올랐다** — AI에게 뭘 어떻게 시켜야 하는지 노하우가 쌓였다
3. **기획서가 없었다** — 그냥 만들면서 결정했다. 기획자가 기획서 없이 만드는 아이러니

3번이 의외로 속도에 큰 영향을 줬다. 기획서를 쓰고, 리뷰하고, 수정하는 과정 없이 "이렇게 하면 어떨까?" → 바로 코드로 → 화면에서 확인 → "아니다, 이쪽이 낫겠다" → 바로 수정. 이 사이클이 분 단위로 돌았다.

참고로 이 프로젝트의 **모든 커밋이 AI(Claude)와의 협업**이다. 100% 바이브 코딩. 내가 "맞고 규칙에서 쪽이 발생하는 조건을 구현해줘"라고 하면 AI가 코드를 짜고, 내가 게임을 돌려보면서 "쪽일 때 상대방 피 한 장을 뺏어야 하는데 안 빠지고 있어"라고 피드백하고, AI가 수정하는 흐름이다.

## 光끼의 맞고 — 게임 소개

앱 이름은 **"光끼의 맞고"**다. 光(빛 광)은 화투의 광(5광, 비광 할 때 그 광)이고, 끼는 "재주" 또는 "기운"이다.

```
┌─────────────────────────────────────┐
│          光끼의 맞고                   │
│  "순시미와 고냥이와 함께하는             │
│         즐거운 맞고 한판!"              │
├─────────────────────────────────────┤
│                                     │
│  👤 상대방 영역 (22%)                  │
│  ┌───────────────────────────────┐  │
│  │ 아바타 | 획득 카드 | 점수       │  │
│  └───────────────────────────────┘  │
│                                     │
│  🃏 바닥 영역 (46%)                   │
│  ┌───────────────────────────────┐  │
│  │ 바닥 카드  |  덱  |  뻑 더미    │  │
│  └───────────────────────────────┘  │
│                                     │
│  🙋 내 영역 (32%)                    │
│  ┌───────────────────────────────┐  │
│  │ 내 패  |  내 획득 카드  | 조작   │  │
│  └───────────────────────────────┘  │
│                                     │
└─────────────────────────────────────┘
```

화면은 3개 영역으로 나뉜다. 상대방 영역, 바닥(필드), 내 영역. 순시미네에서 4탭 구조를 만들 때는 "사용자 동선"을 고민했는데, 게임에서는 "시선 흐름"을 고민해야 했다. 카드 게임은 바닥을 가장 많이 보니까 화면 중앙에 46%를 배정했다.

캐릭터는 방장이 **순시미**, 참가자가 **고냥이**다. 각 캐릭터는 상황에 따라 표정이 바뀐다:

| 상태 | 설명 |
|------|------|
| normal | 기본 표정 |
| 4turn | 남은 턴이 4턴 이하 (초조한 표정) |
| win | 이겼을 때 |
| lose | 졌을 때 |
| mad | 광끼 게이지가 찼을 때 |

디자이너가 각 상태별 아바타를 그려줬다. 순시미네 프로젝트에서부터 함께한 디자이너인데, 이번엔 화투 카드 48장의 에셋까지 준비해줬다. 디자이너 없이는 이 프로젝트가 불가능했을 거다.

## 이번 편을 마치며

맞고 게임을 만들겠다고 결심한 건, 순시미네와 백엔드 리팩토링을 거치면서 "나도 뭔가 만들 수 있다"는 자신감이 있었기 때문이다. 근데 업무용 서비스와 게임은 완전히 다른 세계였다.

서비스 앱은 데이터를 보여주고 저장하는 게 핵심이다. 게임은 **규칙을 코드로 번역하는 게** 핵심이다. 맞고의 규칙이 얼마나 복잡하고, 그걸 코드로 옮기는 게 얼마나 정신없는 일인지는 다음 편에서 이야기하겠다.
