# 레거시 앱에 쿠폰 시스템을 심다

## 갑자기 쿠폰이 필요해졌다

7편까지 매칭, 돌봄, 푸시, 알림톡을 정비했다. 서비스의 핵심 흐름은 어느 정도 갖춰진 상태. 그런데 사업팀에서 새로운 요구가 왔다.

> "제휴 병원 쿠폰을 회원한테 발급하고 싶은데, 어드민에서 관리할 수 있게 해주세요."

배경은 이렇다. 우리 서비스는 돌봄 매칭 플랫폼이다. 보호자가 어르신을 위해 케어매니저를 찾는 서비스인데, 여기에 제휴 병원 할인 쿠폰을 붙이면 사용자 유입과 병원 제휴 영업 양쪽에 도움이 된다. "순시미네 앱에서 이 쿠폰 받으면 제휴 병원에서 할인받을 수 있어요" — 이런 흐름이다.

기획자 입장에서 쿠폰은 익숙한 개념이다. 근데 기존 앱에 쿠폰 시스템을 **처음부터** 심는 건 생각보다 고려할 게 많았다.

## 기존 구조 파악 — 있는 것과 없는 것

먼저 현재 시스템에 쿠폰 관련으로 뭐가 있는지 살펴봤다.

### 있는 것

DB에 이미 두 개의 테이블이 있었다.

| 테이블 | 역할 | 주요 필드 |
|--------|------|-----------|
| `CouponList` | 쿠폰 마스터 목록 | couponName, hospitalName, validityDays, totalQuantity, remainingQuantity |
| `IssuedCoupon` | 회원에게 발급된 쿠폰 | memberId, couponId, couponCode, issuedAt, expireAt, usedYn |

그리고 API도 일부 있었다.

| 엔드포인트 | Method | 설명 |
|-----------|--------|------|
| `/v2/coupons` | GET | 쿠폰 목록 조회 |
| `/v2/coupons/issued` | GET | 발급된 쿠폰 목록 |
| `/v2/coupons/issue` | POST | 쿠폰 발급(다운로드) |
| `/v2/coupons/use` | POST | 쿠폰 사용 처리 |

앱에서 회원이 쿠폰 목록을 보고, 발급받고, 사용하는 기본 흐름은 이미 구현되어 있었던 거다.

### 없는 것

문제는 **관리자 측 기능이 전혀 없다**는 것이었다.

```
쿠폰 등록   → POST /v2/coupons      ❌ 없음
쿠폰 수정   → PUT /v2/coupons/{id}   ❌ 없음
쿠폰 삭제   → DELETE /v2/coupons/{id} ❌ 없음
```

쿠폰 마스터 데이터를 등록하고 수정하고 삭제하는 API가 하나도 없었다. 지금까지는 DB에 직접 INSERT를 해서 쿠폰을 넣고 있었던 것이다. 어드민 화면은 당연히 없고.

그리고 한 가지 더. **쿠폰 게시기간**이라는 개념이 아예 빠져 있었다. 쿠폰을 등록하면 영원히 노출되는 구조. "3월 한 달만 발급 가능한 프로모션 쿠폰" 같은 건 구현이 불가능했다.

정리하면 이번에 해야 할 일은 세 가지였다.

1. **DB 스키마에 게시기간 컬럼 추가**
2. **누락된 API 3개(등록/수정/삭제) 신규 개발**
3. **어드민에 쿠폰 관리 UI 추가** (다음 편)

## 게시기간이라는 개념

쿠폰에는 두 가지 시간 개념이 필요하다. 헷갈리기 쉬운데 정리하면 이렇다.

| 개념 | 의미 | 저장 위치 |
|------|------|-----------|
| **게시기간** | 쿠폰을 발급받을 수 있는 기간 | `CouponList.postingStartAt ~ postingEndAt` |
| **유효기간** | 발급받은 쿠폰을 사용할 수 있는 기간 | `IssuedCoupon.expireAt` (발급일 + validityDays) |

예를 들어, "3월 1일~31일에만 받을 수 있고, 받은 날로부터 30일 안에 써야 하는 쿠폰"이라면:
- 게시기간: 3월 1일 ~ 3월 31일
- 유효기간: 발급일 + 30일 (3월 15일에 받았으면 4월 14일까지)

게시기간이 지나면 앱에서 쿠폰이 아예 보이지 않아야 한다. 수량이 남아 있든, 유효기간이 남아 있든 상관없이. 이건 기획적으로 중요한 포인트였다. 프로모션이 끝났는데 쿠폰이 계속 노출되면 안 되니까.

### DB 스키마 변경

`CouponList` 테이블에 두 개의 컬럼을 추가했다.

```sql
ALTER TABLE coupon_list
  ADD COLUMN posting_start_at TIMESTAMP NULL AFTER coupon_image_url,
  ADD COLUMN posting_end_at TIMESTAMP NULL AFTER posting_start_at;
```

Prisma 스키마에도 반영:

```prisma
model CouponList {
  couponId         BigInt    @id @default(autoincrement()) @map("coupon_id")
  couponName       String    @map("coupon_name")
  hospitalName     String    @map("hospital_name")
  couponDescription String?  @map("coupon_description") @db.Text
  validityDays     Int       @map("validity_days")
  totalQuantity    Int       @map("total_quantity")
  remainingQuantity Int      @map("remaining_quantity")
  couponImageUrl   String?   @map("coupon_image_url")
  postingStartAt   DateTime? @map("posting_start_at")
  postingEndAt     DateTime? @map("posting_end_at")
  // ...
}
```

NULL을 허용한 이유는 기존 데이터 때문이다. 이미 등록된 쿠폰은 게시기간이 없으니까. NULL이면 "무기한 게시"로 처리한다.

## 기존 목록 조회 API 수정

게시기간 컬럼을 추가했으니, 기존 목록 조회 API(`GET /v2/coupons`)도 수정해야 했다. 게시기간이 끝난 쿠폰은 목록에서 빠져야 하니까.

```typescript
// GET /v2/coupons — 쿠폰 목록 조회
const now = new Date();

const coupons = await prisma.couponList.findMany({
  where: {
    isDeleted: false,
    OR: [
      // 게시기간이 설정되지 않은 쿠폰 (무기한)
      { postingEndAt: null },
      // 게시기간 내인 쿠폰
      {
        postingStartAt: { lte: now },
        postingEndAt: { gte: now },
      },
    ],
  },
  orderBy: { createdAt: "desc" },
});
```

여기서 고민이 있었다. `postingStartAt`이 아직 안 된 쿠폰(미래에 게시 시작)도 숨겨야 하는가? 결론은 **앱 API에서는 숨기고, 어드민 API에서는 전부 보여준다**로 정했다. 관리자는 아직 게시 전인 쿠폰도 확인하고 수정할 수 있어야 하니까.

## 누락된 API 3개 — 등록, 수정, 삭제

### 쿠폰 등록 API

```typescript
// POST /v2/coupons — 쿠폰 등록
export async function POST(request: NextRequest) {
  const body = await request.json();

  const coupon = await prisma.couponList.create({
    data: {
      couponName: body.couponName,
      hospitalName: body.hospitalName,
      couponDescription: body.couponDescription,
      validityDays: body.validityDays,
      totalQuantity: body.totalQuantity,
      remainingQuantity: body.totalQuantity, // 처음엔 전체 수량 = 잔여 수량
      couponImageUrl: body.couponImageUrl,
      postingStartAt: body.postingStartAt ? new Date(body.postingStartAt) : null,
      postingEndAt: body.postingEndAt ? new Date(body.postingEndAt) : null,
    },
  });

  return success(coupon);
}
```

`remainingQuantity`를 `totalQuantity`와 동일하게 초기화하는 부분이 포인트다. 쿠폰이 발급될 때마다 `remainingQuantity`가 1씩 줄어드는 구조. 잔여 수량이 0이 되면 더 이상 발급할 수 없다.

`couponDescription`은 `@db.Text` 타입으로 설정했다. 쿠폰 사용 가이드를 Tiptap 에디터로 작성하면 HTML 태그가 포함된 긴 텍스트가 들어오기 때문이다. VARCHAR(255)로는 부족하다.

### 쿠폰 수정 API

```typescript
// PUT /v2/coupons/{id} — 쿠폰 수정
export async function PUT(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const couponId = BigInt(params.id);
  const body = await request.json();

  // 발급된 미사용 쿠폰이 있는지 확인
  const activeIssued = await prisma.issuedCoupon.findFirst({
    where: {
      couponId,
      usedYn: false,
    },
  });

  if (activeIssued) {
    return error(400, "이미 대상 쿠폰을 발급 받은 회원이 존재하여 수정이 불가능합니다.");
  }

  const coupon = await prisma.couponList.update({
    where: { couponId },
    data: {
      couponName: body.couponName,
      hospitalName: body.hospitalName,
      couponDescription: body.couponDescription,
      validityDays: body.validityDays,
      totalQuantity: body.totalQuantity,
      remainingQuantity: body.totalQuantity,
      couponImageUrl: body.couponImageUrl,
      postingStartAt: body.postingStartAt ? new Date(body.postingStartAt) : null,
      postingEndAt: body.postingEndAt ? new Date(body.postingEndAt) : null,
    },
  });

  return success(coupon);
}
```

여기서 핵심은 **수정 전 검증**이다. `IssuedCoupon` 테이블에서 해당 쿠폰을 발급받았고 아직 사용하지 않은(`usedYn: false`) 회원이 한 명이라도 있으면 수정을 막는다. 이미 받아간 쿠폰의 조건이 바뀌면 혼란이 생기니까.

단, 발급받은 회원이 **전부 사용 완료**한 상태(`usedYn: true`)라면 수정이 가능하다. 이미 쓴 쿠폰은 조건이 바뀌어도 영향이 없으니까.

이 검증 로직은 기획서에 "이미 대상 쿠폰을 발급 받은 회원이 존재하여 수정 또는 삭제가 불가능합니다."라고 한 줄로 적혀 있었는데, 코드로 만들면 "발급 받았지만 사용 완료한 경우는 예외"라는 엣지 케이스까지 고려해야 한다. 기획서 한 줄이 코드 수십 줄이 되는 전형적인 예다.

### 쿠폰 삭제 API

```typescript
// DELETE /v2/coupons/{id} — 쿠폰 삭제
export async function DELETE(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const couponId = BigInt(params.id);

  // 수정과 동일한 검증
  const activeIssued = await prisma.issuedCoupon.findFirst({
    where: {
      couponId,
      usedYn: false,
    },
  });

  if (activeIssued) {
    return error(400, "이미 대상 쿠폰을 발급 받은 회원이 존재하여 삭제가 불가능합니다.");
  }

  await prisma.couponList.update({
    where: { couponId },
    data: { isDeleted: true },
  });

  return success({ deleted: true });
}
```

물리 삭제가 아니라 **soft delete**다. `isDeleted: true`로 표시만 한다. 4편에서 돌봄일지 댓글에 적용한 것과 같은 전략이다. 쿠폰 발급 이력이 `IssuedCoupon`에 남아 있는데, 마스터 데이터를 물리 삭제하면 참조 무결성이 깨진다.

## 쿠폰 발급 흐름 전체 정리

전체 흐름을 그려보면 이렇다.

```
[관리자 — 어드민]
   │
   ├─ 쿠폰 등록 (POST /v2/coupons)
   │    → CouponList 레코드 생성
   │    → 게시기간, 유효기간(일), 수량, 이미지, 설명 설정
   │
   ├─ 쿠폰 수정 (PUT /v2/coupons/{id})
   │    → 미사용 발급 건 없을 때만 가능
   │
   └─ 쿠폰 삭제 (DELETE /v2/coupons/{id})
        → 미사용 발급 건 없을 때만 가능 (soft delete)

[회원 — 앱]
   │
   ├─ 쿠폰 목록 조회 (GET /v2/coupons)
   │    → 게시기간 내 + 잔여수량 > 0인 쿠폰만 표시
   │
   ├─ 쿠폰 발급 (POST /v2/coupons/issue)
   │    → IssuedCoupon 레코드 생성
   │    → couponCode 자동 생성
   │    → expireAt = 발급일 + validityDays
   │    → remainingQuantity -= 1
   │
   └─ 쿠폰 사용 (POST /v2/coupons/use)
        → usedYn = true, usedAt = now
        → 제휴 병원에서 couponCode 확인 후 할인 적용
```

관리자와 회원의 흐름이 깔끔하게 분리된다. 관리자는 쿠폰 마스터를 관리하고, 회원은 발급과 사용만 한다. 두 흐름이 만나는 접점은 `CouponList` 테이블 하나뿐이다.

## 이미지와 설명 — 쿠폰도 콘텐츠다

쿠폰 등록에서 의외로 신경 쓸 부분이 두 가지 있었다.

### 쿠폰 이미지

쿠폰에 이미지를 붙일 수 있다. 제휴 병원 로고나 혜택을 시각적으로 보여주기 위해서. 이미지 업로드는 기존 S3 업로드 API를 재사용했다.

```
POST /v1/image/upload
  → attachedType: "COUPON_IMAGE"
  → S3에 업로드 → URL 반환
  → 반환된 URL을 couponImageUrl에 저장
```

`AttachedFile` 테이블에 메타데이터가 기록되고, `attachedType`으로 쿠폰 이미지를 구분한다. 기존에 프로필, 자격증, 돌봄일지 등의 이미지를 같은 구조로 관리하고 있어서 새로 만들 건 없었다. `COUPON_IMAGE` 타입만 추가하면 됐다.

### 쿠폰 설명 (사용 가이드)

이게 좀 까다로웠다. 기획 요구사항이 이랬다:

> "쿠폰 사용가이드는 블로그 글 입력처럼 어드민에서 입력한 텍스트 입력 및 줄바꿈 등에 따라 웹 프론트에도 동일하게 적용되어야 함"

단순 textarea로는 안 된다. 줄바꿈, 볼드, 글자 크기 조정 등이 가능해야 한다. 다행히 어드민에 이미 블로그 작성용 Tiptap 에디터가 구현되어 있었다. 이걸 재사용하면 된다.

Tiptap 에디터로 작성하면 HTML 형식으로 저장되고, 앱 프론트에서 그대로 렌더링한다. `couponDescription` 컬럼을 TEXT 타입으로 잡은 이유가 이것이다. HTML 태그가 포함된 리치 텍스트가 들어가니까.

## 이번 편을 마치며

이번 편에서는 쿠폰 시스템의 **백엔드 기반**을 다졌다.

1. 게시기간이라는 새 개념을 DB에 반영했고
2. 누락된 CRUD API 3개를 새로 만들었고
3. 수정/삭제 시 발급된 쿠폰 검증 로직을 넣었다

기획서에는 "쿠폰 관리 기능 추가"라고 한 줄이지만, 실제로는 DB 스키마 변경, 기존 API 수정, 신규 API 3개, 검증 로직, 이미지 타입 추가, 에디터 연동까지 — 생각보다 건드릴 게 많았다.

그래도 이전 편들에서 다진 Prisma + Next.js API Routes 기반이 있으니 작업 자체는 빠르게 진행됐다. 패턴이 잡혀 있으면 새 기능을 추가하는 게 훨씬 수월하다.

다음 편에서는 이 API들을 어드민 화면에서 사용할 수 있도록 프론트엔드를 만든다. React Query로 API를 연결하고, Tiptap 에디터로 쿠폰 설명을 작성하고, 발급현황을 모니터링하는 관리자 화면을 올린다.
