# 어드민에 햄릿 관리와 지도를 올리다

## 햄릿이 뭔데

"햄릿"이라고 하면 셰익스피어가 떠오르겠지만, 우리 서비스에서 햄릿은 **행정구역 기반의 돌봄 구역 단위**다. 요양보호사와 어르신을 매칭할 때 "이 동네는 이 햄릿에 속한다"는 기준이 필요한데, 그 구역을 정의하고 관리하는 기능이 어드민에 없었다.

지금까지는 DB에 직접 들어가서 햄릿 데이터를 넣거나 수정했다. 운영팀이 "이 동 추가해주세요"라고 요청하면, 개발자가 DB 콘솔을 열어서 INSERT 쿼리를 날리는 식이었다. 당연히 비효율적이고, 실수 여지도 크다.

이번에 만든 건 세 가지다.

1. **햄릿 목록/상세/등록** — 기본 CRUD
2. **개별 햄릿 구역 지도** — 법정동 경계를 Leaflet으로 시각화
3. **전체 햄릿 지도** — DB에 등록된 모든 햄릿의 경계를 한 지도에 통합 표시

특히 지도가 핵심이다. 햄릿이 어떤 동을 포함하는지 텍스트로만 보면 감이 안 온다. 지도에 경계선을 그려주면 운영팀이 "아, 이 구역이 이 범위구나"를 한눈에 파악할 수 있다.

## 기존 패턴 위에 올리기

[[백엔드 도전기 9편 - 어드민에 쿠폰 관리 화면을 올리다|9편]]에서 쿠폰 관리 화면을 만들 때 얘기했듯이, 어드민에서 새 기능을 추가하는 가장 빠른 방법은 **기존 패턴을 복제해서 수정하는 것**이다.

햄릿 관리도 같은 패턴을 따랐다. 사이드바 메뉴 추가, 타입 정의, TanStack Query 훅, DataTable 컬럼 — 쿠폰 때 만들어둔 구조를 그대로 가져다 썼다.

```
app/(dashboard)/hamlet/
├── page.tsx                    # 햄릿 목록 (필터, 검색, 페이지네이션, 지도)
├── columns.tsx                 # DataTable 컬럼 정의
├── types.ts                    # HamletGroup, HamletRow, HamletDetail 타입
├── hooks/useHamletApi.ts       # TanStack Query 훅 모음
├── [hamletCode]/page.tsx       # 햄릿 상세 (정보 수정, 행 추가/삭제, 지도)
└── add/page.tsx                # 햄릿 등록
```

다만 API는 신규 개발이 필요했다.

| 메서드    | 엔드포인트                          | 용도          |
| ------ | ------------------------------ | ----------- |
| GET    | `/v2/admin/hamlet`             | 목록 (페이지네이션) |
| GET    | `/v2/admin/hamlet/{code}`      | 상세          |
| POST   | `/v2/admin/hamlet`             | 등록          |
| PUT    | `/v2/admin/hamlet/{code}`      | 이름 수정       |
| DELETE | `/v2/admin/hamlet/{code}`      | 삭제          |
| POST   | `/v2/admin/hamlet/{code}/rows` | 행 추가        |
| DELETE | `/v2/admin/hamlet/row/{id}`    | 행 삭제        |
| GET    | `/v2/admin/hamlet/filters`     | 필터 옵션       |
| GET    | `/v2/admin/hamlet/csv`         | CSV 다운로드    |

CRUD 자체는 쿠폰 때와 크게 다르지 않다. 필터 검색, 페이지네이션, CSV 다운로드, 인라인 편집, AlertDialog 삭제 확인 — 전부 이전에 해본 패턴이다. React Query의 `invalidateQueries`로 등록/수정/삭제 후 목록을 자동 갱신하는 것도 동일하다.

재미있는 건 햄릿 등록 시 **같은 시/군/구에 기존 햄릿이 있으면 추천 다이얼로그를 띄우는 것**이다. 운영팀이 중복 등록하는 실수를 방지하기 위해 넣은 건데, 이건 백엔드의 `/v2/admin/hamlet/suggest` API가 해준다. 프론트에서는 다이얼로그만 띄우면 된다.

CRUD까지는 순조로웠다. 문제는 지도에서 터졌다.

## Leaflet으로 지도 올리기 — SSR의 함정

햄릿 구역을 시각화하려면 지도 라이브러리가 필요하다. **Leaflet**을 골랐다.

Google Maps나 Naver Maps 같은 상용 지도 SDK도 있지만, 우리가 필요한 건 "법정동 경계 폴리곤을 지도 위에 그리는 것"이다. 배경 지도 타일 위에 커스텀 폴리곤을 자유롭게 그릴 수 있으면 충분하고, Leaflet이 딱 그 용도다. 무료이고, 가볍고, React 바인딩(`react-leaflet`)도 잘 되어 있다.

그런데 Next.js에서 Leaflet을 쓰면 한 가지 함정이 있다. **Leaflet은 `window` 객체에 의존한다.** 서버 사이드 렌더링(SSR) 환경에서는 `window`가 없으니, 컴포넌트를 import하는 순간 에러가 난다.

해결법은 `dynamic import`와 `ssr: false` 조합이다.

```typescript
const HamletMap = dynamic(
  () => import("@/components/hamlet/HamletMap"),
  { ssr: false }
);
```

이렇게 하면 Leaflet 컴포넌트가 클라이언트 사이드에서만 로드된다. Leaflet의 CSS도 동적으로 head에 추가해야 한다. 이건 Next.js + Leaflet 조합에서 거의 필수적인 보일러플레이트인데, 한번 해두면 다른 지도 컴포넌트에서도 그대로 쓸 수 있다.

지도 컴포넌트는 두 개를 만들었다.

- **HamletMap** — 개별 햄릿의 경계를 보여주는 지도. 상세 페이지에서 사용
- **AllHamletsMap** — DB에 등록된 전체 햄릿의 경계를 한 지도에 통합 표시. 목록 페이지에서 다이얼로그로 열림

여기까지는 괜찮았다. 진짜 문제는 **경계 데이터를 어디서 가져오느냐**였다.

## VWorld API — 법정동 경계 폴리곤 가져오기

햄릿에 속한 동의 경계를 지도에 그리려면 **법정동 경계 폴리곤 데이터**가 필요하다. 이걸 제공하는 게 **VWorld**(브이월드), 국토교통부가 운영하는 공간정보 오픈 API다.

```
GET https://api.vworld.kr/req/data
  ?service=data
  &request=GetFeature
  &data=LT_C_ADEMD_INFO          ← 읍면동 행정구역 데이터
  &attrFilter=emd_cd:=:{bcode}   ← 법정동코드로 필터
  &geometry=true                   ← 경계 폴리곤 포함
  &format=json
  &crs=EPSG:4326
  &key={API_KEY}
```

법정동코드(bcode)를 넘기면 해당 동의 경계 좌표가 GeoJSON 형태로 돌아온다. 이걸 Leaflet의 `Polygon`으로 렌더링하면 지도에 경계선이 그려진다.

문제는 VWorld API를 클라이언트(브라우저)에서 직접 호출하면 **CORS 에러**가 난다는 거다. 정부 API가 브라우저 요청의 `Origin` 헤더를 허용하지 않는다. 그래서 Next.js API Route를 프록시로 사용했다.

```
브라우저 → /api/vworld (Next.js API Route) → api.vworld.kr
```

클라이언트는 `/api/vworld`로 요청하고, 서버 사이드에서 VWorld API를 호출한 뒤 결과를 그대로 반환한다. CORS 우회의 정석적인 방법이다.

로컬에서 테스트했다. **잘 된다.** 법정동 경계가 지도 위에 깔끔하게 그려진다. 여기까지는 순탄했다.

## 배포하면 502 — Vercel 리전의 함정

로컬에서 다 되는 걸 확인하고, dev 환경에 배포했다. 그리고 지도를 열었다.

**502 에러.**

경계 폴리곤이 안 나온다. 지도 타일은 잘 뜨는데, VWorld API만 실패한다.

여기서부터 디버깅 삽질이 시작됐다.

### 1차 의심: Rate Limit?

VWorld 관리 페이지에 로그인해서 확인했다. "일일제한: 무제한, 오늘 접속: 62건." Rate limit이 아니다.

### 2차 의심: API 키?

Vercel 환경변수에 `NEXT_PUBLIC_VWORLD_API_KEY`가 제대로 설정되어 있나 확인했다. 정상이다. 같은 키로 로컬 터미널에서 `curl`을 날려봤다. 정상 응답이 온다. API 키 문제도 아니다.

### 3차 시도: 에러 상세 추가

API Route에 에러 상세 로깅을 추가해서 배포했다. VWorld가 뭐라고 응답하는지 확인하려고. 결과는... VWorld 서버 자체가 응답을 안 주는 것 같았다. 타임아웃.

### 4차 시도: preferredRegion 설정

혹시 Vercel 서버리스 함수의 위치 문제인가 싶어서, Next.js의 `preferredRegion` export를 추가했다.

```typescript
// app/api/vworld/route.ts
export const preferredRegion = ["icn1", "hnd1"]; // 서울, 도쿄
```

배포. **효과 없음.** 여전히 502.

### 5차: Vercel 로그 분석 — 범인 발견

Vercel 대시보드에서 함수 실행 로그를 꼼꼼히 뜯어봤다. 그리고 결정적인 두 줄을 발견했다.

```
Received in Seoul, South Korea (icn1)     ← 요청 수신은 서울
Routed to Washington, D.C., USA (iad1)    ← 함수 실행은 미국!
```

**요청은 서울에서 받았지만, 실제 서버리스 함수는 미국(워싱턴 D.C.)에서 실행되고 있었다.**

VWorld는 한국 정부 API다. **해외 IP에서의 요청을 차단한다.** 미국에 있는 Vercel 서버가 VWorld에 요청을 보내니, VWorld 입장에서는 해외 IP니까 거부한 거다.

로컬에서는 당연히 한국 IP로 요청이 가니까 잘 됐고, 배포 환경에서만 터진 이유가 이거였다.

### `preferredRegion`은 왜 안 먹혔나

여기서 중요한 교훈이 하나 있다. Next.js의 `preferredRegion` export와 `vercel.json`의 `functions.regions` 설정은 **다른 것**이다.

> `preferredRegion`은 **요청 수신 위치**만 변경한다. 실제 함수 실행 리전은 변경하지 않는다. `vercel.json`의 `functions.regions`를 사용해야 **실행 리전**이 변경된다.

이걸 몰랐다. Vercel 로그에 "Received in Seoul"이라고 찍힌 게 `preferredRegion` 덕분이었는데, 정작 함수가 돌아가는 건 미국이었다. 수신 위치와 실행 위치가 다를 수 있다는 걸 처음 알았다.

### 해결: vercel.json으로 실행 리전 고정

```json
{
  "functions": {
    "app/api/vworld/route.ts": {
      "maxDuration": 10,
      "memory": 256,
      "regions": ["icn1"]
    }
  }
}
```

`vercel.json`에서 VWorld API Route 함수의 **실행 리전을 서울(icn1)로 고정**했다. 배포 후 확인 — **정상 동작.** 경계 폴리곤이 지도에 깨끗하게 그려진다.

커밋 7개 중 4개가 이 디버깅 과정이었다. `feat` 2개, `fix` 3개, `debug` 1개. 기능을 만드는 것보다 배포 환경 이슈를 잡는 데 더 많은 커밋이 쓰였다.

## 전체 햄릿 지도 — 210건의 API 호출

개별 햄릿 지도가 되니까 욕심이 났다. **"DB에 등록된 전체 햄릿의 경계를 한 지도에 보여주면 어떨까?"**

운영팀 입장에서 전체 서비스 커버리지를 한눈에 보는 건 꽤 유용하다. "우리가 어디까지 커버하고 있지?"를 지도 한 장으로 확인할 수 있으니까.

목록 페이지에 [햄릿 지도] 버튼을 추가하고, 누르면 다이얼로그에서 `AllHamletsMap` 컴포넌트가 열리게 했다. 다이얼로그가 열릴 때만 데이터를 로딩하는 lazy 로딩이다.

그런데 문제가 생겼다. **전체 햄릿에 속한 법정동 코드가 210개 이상**이었다. 즉, VWorld API를 210번 이상 호출해야 전체 경계를 그릴 수 있다.

처음에는 5개씩 배치로 나눠서 호출했는데, 210건이 한꺼번에 쏟아지니 **무한 로딩** 상태에 빠졌다.

해결은 **방어적 호출 전략**이다.

| 설정 | 값 | 이유 |
|------|-----|------|
| 최대 요청 수 | 50건 | 초과분은 생략 + 사용자에게 안내 |
| 배치 크기 | 2건씩 | 동시 요청 최소화 |
| 배치 간 딜레이 | 500ms | API 서버 부하 방지 |
| 개별 타임아웃 | 6초 | 응답 없으면 스킵 |
| 조기 중단 | 연속 5회 실패 | 서버 이상 시 빠르게 포기 |

50건 제한이 좀 아쉽긴 하다. 전체 210건을 다 그리면 더 완전한 지도가 되겠지만, 현실적으로 API 호출 210번은 사용자 경험이 좋지 않다. 향후에는 VWorld 응답을 DB에 캐싱하는 게 맞다. 법정동 경계는 자주 바뀌는 데이터가 아니니까.

전체 지도에서 햄릿별로 색상을 구분하기 위해 16색 팔레트를 쓰고, 우하단에 범례(legend)도 추가했다. 어드민 화면이지만 이런 시각적 디테일은 운영팀의 사용성에 큰 차이를 만든다.

## 결과물 정리

하루 동안 만든 결과를 정리하면 이렇다.

| 구분 | 파일 수 | 내용 |
|------|---------|------|
| 햄릿 페이지 | 6개 | 목록, 상세, 등록, 컬럼, 타입, 훅 |
| 지도 컴포넌트 | 4개 | 개별 지도, 전체 지도, 주소 검색, 추천 다이얼로그 |
| API Route | 2개 | VWorld 프록시, 주소 API 콜백 |

의존성도 3개 추가했다. `leaflet`, `react-leaflet`, `@types/leaflet`. Leaflet은 SSR 불가니까 반드시 `dynamic import + ssr: false`로 쓴다 — 이건 Next.js에서 Leaflet을 쓰는 사람이라면 무조건 만나는 이슈다.

## 돌아보며 — "로컬에서 되는데 배포에서 안 돼요"

이번 작업에서 가장 크게 배운 건 VWorld 디버깅이다. 정확히 말하면 **"Vercel 서버리스 함수의 실행 리전은 내가 생각하는 곳이 아닐 수 있다"**는 교훈이다.

[[맞고 3편 - 실시간 대전에서 Vercel 배포까지|맞고 3편]]에서도 Vercel 배포를 다뤘지만, 그때는 Flutter Web의 정적 호스팅이라 리전 문제가 없었다. 이번에는 **API Route(서버리스 함수)**를 통해 한국 정부 API를 호출하면서, 함수가 어디서 실행되느냐가 성패를 갈랐다.

한국 정부 API를 Vercel에서 프록시할 때 체크리스트를 정리해두면:

1. **한국 정부 API는 대부분 해외 IP를 차단한다** — VWorld, 공공데이터포털 등
2. **`preferredRegion` ≠ 실행 리전** — 수신 위치만 바뀌고 실행은 기본 리전(미국)
3. **`vercel.json`의 `functions.regions`를 써야 실행 리전이 바뀐다**
4. **Vercel 로그에서 "Received in" vs "Routed to"를 반드시 확인**한다

솔직히 CRUD 기능 자체는 쿠폰 때 만들어둔 패턴 위에 올리는 거라 반나절이면 충분했다. 나머지 반나절을 VWorld 디버깅에 쓴 셈이다. 하지만 이 삽질 덕분에 Vercel의 리전 구조를 확실히 이해하게 됐다. 다음에 한국 정부 API를 연동할 일이 있으면 `vercel.json`부터 먼저 세팅할 거다.

9편에서도 얘기했지만, **기존 패턴이 잡혀 있으면 새 기능 추가가 빠르다.** 쿠폰 관리에서 만든 DataTable, FilterBar, React Query 훅 패턴을 그대로 가져다 쓰니, 햄릿 CRUD는 정말 빠르게 끝났다. 레거시 리팩토링의 진짜 가치는 이런 데서 나온다 — 패턴이 한번 잡히면 그 위에 쌓는 건 훨씬 쉬워진다.
