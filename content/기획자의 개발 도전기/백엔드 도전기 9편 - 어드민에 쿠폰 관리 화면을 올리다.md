# 어드민에 쿠폰 관리 화면을 올리다

## 백엔드는 됐고, 이제 화면이다

[[백엔드 도전기 8편 - 레거시 앱에 쿠폰 시스템을 심다|8편]]에서 쿠폰 CRUD API와 DB 스키마를 만들었다. 이제 관리자가 실제로 쿠폰을 등록하고, 수정하고, 발급현황을 확인할 수 있는 어드민 화면을 올려야 한다.

우리 어드민(`ssm-admin`)은 Next.js + Shadcn UI 기반이다. 매칭 승인, 회원 관리, 블로그 관리 등의 기능이 이미 들어가 있어서, 기존 패턴을 따라가면 된다. 새 기능을 추가할 때 제일 편한 건 **잘 만들어진 기존 페이지를 복제해서 수정하는 것**이다.

## 사이드바 메뉴 추가

먼저 좌측 사이드바에 "쿠폰 관리" 메뉴를 추가했다.

```typescript
// config/menus.ts
{ isHeader: true, title: "쿠폰 관리" },
{
  title: "쿠폰 등록하기",
  href: "/coupon/list",
  icon: "heroicons:ticket",
},
{
  title: "쿠폰 발급현황",
  href: "/coupon/issued",
  icon: "heroicons:clipboard-document-list",

```

두 개의 하위 메뉴로 구성했다. "쿠폰 등록하기"는 쿠폰 마스터 데이터를 관리하는 CRUD 페이지이고, "쿠폰 발급현황"은 회원별 발급/사용 상태를 모니터링하는 페이지다.

## 타입 정의 — 프론트와 백엔드의 계약

API 연동 전에 타입부터 정의했다. TypeScript 프로젝트에서 타입은 프론트와 백엔드 사이의 계약서 같은 거다.

```typescript
// app/(dashboard)/coupon/types.ts

// 쿠폰 마스터 (CouponList 테이블)
export interface Coupon {
  coupon_id: number;
  couponName: string;
  hospitalName: string;
  couponDescription: string | null;
  validityDays: number;
  totalQuantity: number;
  remainingQuantity: number;
  couponImageUrl: string | null;
  attachedFileKey: string | null;
  postingStartAt: string | null;
  postingEndAt: string | null;
  isActive: boolean;
  createdAt: string;
  updatedAt: string;
}

// 발급된 쿠폰 (IssuedCoupon 테이블)
export interface IssuedCoupon {
  issuedCouponId: number;
  memberId: number;
  couponId: number;
  couponCode: string;
  hospitalName: string;
  couponName: string;
  couponDescription: string | null;
  couponImageUrl: string | null;
  issuedAt: string;
  expireAt: string;
  usedYn: boolean;
  usedAt: string | null;
  updatedAt: string;
}

// 등록/수정 폼용 타입
export interface CouponFormData {
  couponName: string;
  hospitalName: string;
  couponDescription: string;
  validityDays: number;
  totalQuantity: number;
  couponImageUrl: string | null;
  postingStartAt: string | null;
  postingEndAt: string | null;
}
```

`Coupon`과 `IssuedCoupon`은 각각 다른 테이블, 다른 API를 바라본다. `CouponFormData`는 등록/수정 폼에서 사용하는 타입으로, API 응답 타입과 폼 입력 타입을 분리했다.

## React Query 훅 — API와 UI의 연결고리

어드민에서는 React Query(TanStack Query)로 서버 상태를 관리한다. 기존 매칭 승인 페이지의 훅 패턴을 참고해서 쿠폰용 훅을 만들었다.

```typescript
// hooks/useCouponList.ts

export function useCouponList(params: CouponListParams) {
  return useQuery({
    queryKey: ["couponList", params],
    queryFn: async () => {
      const axios = getAxiosInstance("v2");
      const { data } = await axios.get("/coupons", { params });
      return data;
    },
  });
}

export function useCreateCoupon() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async (formData: CouponFormData) => {
      const axios = getAxiosInstance("v2");
      const { data } = await axios.post("/coupons", formData);
      return data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["couponList"] });
      toast.success("쿠폰이 등록되었습니다.");
    },
    onError: (error: any) => {
      toast.error(error.response?.data?.message || "쿠폰 등록에 실패했습니다.");
    },
  });
}

export function useUpdateCoupon() { /* 유사 패턴 */ }
export function useDeleteCoupon() { /* 유사 패턴 */ }
```

```typescript
// hooks/useIssuedCoupon.ts

export function useIssuedCouponList(params: IssuedCouponParams) {
  return useQuery({
    queryKey: ["issuedCouponList", params],
    queryFn: async () => {
      const axios = getAxiosInstance("v2");
      const { data } = await axios.get("/coupons/issued", { params });
      return data;
    },
  });
}
```

`useMutation`의 `onSuccess`에서 `invalidateQueries`로 목록을 자동 갱신하는 게 React Query의 핵심 패턴이다. 쿠폰을 등록하면 목록 페이지의 데이터가 자동으로 다시 조회된다. 수동으로 state를 업데이트하거나 페이지를 새로고침할 필요가 없다.

## 쿠폰 목록 페이지

쿠폰 관리의 첫 화면은 등록된 전체 쿠폰 목록이다.

```typescript
// coupon/list/page.tsx
export default function CouponListPage() {
  const [searchParams, setSearchParams] = useState({
    keyword: "",
    page: 1,
    size: 10,
  });

  const { data, isLoading } = useCouponList(searchParams);

  return (
    <div>
      <PageHeader title="쿠폰 등록하기">
        <Button onClick={() => router.push("/coupon/list/add")}>
          쿠폰등록
        </Button>
      </PageHeader>

      <FilterBar
        onSearch={(keyword) => setSearchParams({ ...searchParams, keyword })}
        placeholder="쿠폰명 또는 병원명으로 검색"
      />

      <DataTable
        columns={columns}
        data={data?.items || []}
        isLoading={isLoading}
        pagination={data?.pagination}
        onPageChange={(page) => setSearchParams({ ...searchParams, page })}
      />
    </div>
  );
}
```

테이블 컬럼은 별도 파일로 분리했다. 어드민의 기존 패턴을 따른 것이다.

```typescript
// coupon/list/columns.tsx
export const columns: ColumnDef<Coupon>[] = [
  { accessorKey: "coupon_id", header: "쿠폰ID" },
  { accessorKey: "couponName", header: "쿠폰명" },
  { accessorKey: "hospitalName", header: "병원명" },
  {
    header: "게시기간",
    cell: ({ row }) => {
      const start = row.original.postingStartAt;
      const end = row.original.postingEndAt;
      if (!start || !end) return "무기한";
      return `${formatDate(start)} ~ ${formatDate(end)}`;
    },
  },
  { accessorKey: "validityDays", header: "유효기간(일)" },
  { accessorKey: "totalQuantity", header: "총수량" },
  { accessorKey: "remainingQuantity", header: "잔여수량" },
  { accessorKey: "createdAt", header: "등록일", cell: formatDateCell },
];
```

게시기간이 NULL이면 "무기한"으로 표시한다. 기존 데이터와의 호환성을 위한 처리다.

## 쿠폰 등록 폼 — Tiptap 에디터의 재활용

쿠폰 등록 페이지가 이번 작업에서 가장 복잡한 부분이었다. 입력 필드가 많고, 특히 쿠폰 설명은 리치 텍스트 에디터가 필요하기 때문이다.

```typescript
// coupon/list/add/page.tsx
export default function CouponAddPage() {
  const form = useForm<CouponFormData>({
    resolver: zodResolver(couponSchema),
    defaultValues: {
      couponName: "",
      hospitalName: "",
      couponDescription: "",
      validityDays: 30,
      totalQuantity: 100,
      couponImageUrl: null,
      postingStartAt: null,
      postingEndAt: null,
    },
  });

  const createCoupon = useCreateCoupon();

  const onSubmit = (data: CouponFormData) => {
    createCoupon.mutate(data, {
      onSuccess: () => router.push("/coupon/list"),
    });
  };

  return (
    <Form {...form}>
      <form onSubmit={form.handleSubmit(onSubmit)}>
        {/* 쿠폰명, 병원명 — 텍스트 인풋 */}
        {/* 게시기간 — DateRangePicker */}
        {/* 유효기간, 수량 — 숫자 인풋 */}
        {/* 쿠폰 이미지 — 이미지 업로더 */}
        {/* 쿠폰 설명 — Tiptap 에디터 */}
      </form>
    </Form>
  );
}
```

React Hook Form + Zod 조합으로 폼 검증을 처리했다. Zod 스키마는 이렇다:

```typescript
const couponSchema = z.object({
  couponName: z.string().min(1, "쿠폰명을 입력해주세요"),
  hospitalName: z.string().min(1, "병원명을 입력해주세요"),
  couponDescription: z.string().optional(),
  validityDays: z.number().min(1, "유효기간은 1일 이상이어야 합니다"),
  totalQuantity: z.number().min(1, "수량은 1개 이상이어야 합니다"),
  couponImageUrl: z.string().nullable(),
  postingStartAt: z.string().nullable(),
  postingEndAt: z.string().nullable(),
});
```

### Tiptap 에디터 — 블로그 작성기 재활용

쿠폰 설명을 위해 기존 블로그 작성용 Tiptap 에디터(`components/blog/TipTapEditor.tsx`)를 그대로 가져다 썼다. 블로그 글을 쓸 때 쓰는 에디터를 쿠폰 설명에도 사용하는 것이다.

```typescript
<TipTapEditor
  content={form.watch("couponDescription")}
  onChange={(html) => form.setValue("couponDescription", html)}
/>
```

이렇게 하면 관리자가 쿠폰 사용 가이드를 작성할 때 글자 크기 조정, 볼드, 줄바꿈 등을 자유롭게 쓸 수 있고, 저장된 HTML이 앱 프론트에서 그대로 렌더링된다. 기획 요구사항이었던 "블로그 글 입력처럼 텍스트 편집 기능 제공"이 에디터 컴포넌트 재활용 한 줄로 해결된 셈이다.

기존에 잘 만들어둔 컴포넌트가 있으면 새 기능 개발이 이렇게 빨라진다. 에디터를 처음부터 만들었으면 이것만으로 며칠이 걸렸을 거다.

### 쿠폰 이미지 업로드

쿠폰 이미지도 기존 이미지 업로드 컴포넌트를 재활용했다.

```typescript
<ImageUploader
  value={form.watch("couponImageUrl")}
  onChange={(url) => form.setValue("couponImageUrl", url)}
  uploadConfig={{
    attachedType: "COUPON_IMAGE",
    maxSize: 10 * 1024 * 1024, // 10MB
    accept: ["image/jpeg", "image/png"],
  }}
/>
```

업로드 흐름은 `POST /v1/image/upload` → S3에 저장 → URL 반환 → `couponImageUrl`에 세팅. 프로필 이미지, 돌봄일지 이미지와 동일한 파이프라인이다.

## 쿠폰 수정 페이지 — 수정과 삭제, 그리고 검증

쿠폰 목록에서 행을 클릭하면 수정 페이지로 이동한다. 등록 폼과 거의 동일하지만 두 가지가 다르다.

**기존 데이터 로드**: 페이지 진입 시 쿠폰 상세 데이터를 API에서 가져와 폼에 채운다.

```typescript
// coupon/list/[id]/mod/page.tsx
const { data: coupon } = useCouponDetail(params.id);

useEffect(() => {
  if (coupon) {
    form.reset({
      couponName: coupon.couponName,
      hospitalName: coupon.hospitalName,
      couponDescription: coupon.couponDescription || "",
      validityDays: coupon.validityDays,
      totalQuantity: coupon.totalQuantity,
      couponImageUrl: coupon.couponImageUrl,
      postingStartAt: coupon.postingStartAt,
      postingEndAt: coupon.postingEndAt,
    });
  }
}, [coupon]);
```

**삭제 버튼 + 검증 얼럿**: 수정 페이지에 삭제 버튼이 함께 있다. 삭제 시 확인 다이얼로그를 띄우고, 만약 미사용 발급 건이 있으면 백엔드에서 400 에러를 반환한다.

```typescript
const deleteCoupon = useDeleteCoupon();

const handleDelete = () => {
  deleteCoupon.mutate(params.id, {
    onSuccess: () => {
      toast.success("쿠폰이 삭제되었습니다.");
      router.push("/coupon/list");
    },
    onError: (error: any) => {
      // 백엔드에서 "발급 받은 회원이 존재하여 삭제가 불가능합니다" 메시지 반환
      toast.error(error.response?.data?.message || "삭제에 실패했습니다.");
    },
  });
};
```

검증 로직이 백엔드에 있기 때문에 프론트에서는 에러 응답을 받아서 메시지만 보여주면 된다. 프론트와 백엔드의 역할 분담이 명확한 것이다. 검증은 백엔드가, 사용자 피드백은 프론트가 담당한다.

## 쿠폰 발급현황 모니터링

두 번째 메뉴인 "쿠폰 발급현황"은 `IssuedCoupon` 테이블을 조회하는 읽기 전용 페이지다. 관리자가 "누가, 어떤 쿠폰을, 언제 받았고, 썼는지"를 한눈에 볼 수 있어야 한다.

```typescript
// coupon/issued/page.tsx
export default function IssuedCouponPage() {
  const [params, setParams] = useState({
    keyword: "",
    usedYn: "ALL", // 전체 | Y | N
    page: 1,
    size: 10,
  });

  const { data, isLoading } = useIssuedCouponList(params);

  return (
    <div>
      <PageHeader title="쿠폰 발급현황" />

      <FilterBar
        onSearch={(keyword) => setParams({ ...params, keyword })}
        placeholder="쿠폰명, 회원ID, 쿠폰코드로 검색"
      >
        <Select
          value={params.usedYn}
          onValueChange={(value) => setParams({ ...params, usedYn: value })}
        >
          <SelectItem value="ALL">전체</SelectItem>
          <SelectItem value="Y">사용완료</SelectItem>
          <SelectItem value="N">미사용</SelectItem>
        </Select>
      </FilterBar>

      <DataTable
        columns={issuedColumns}
        data={data?.items || []}
        isLoading={isLoading}
        pagination={data?.pagination}
        onPageChange={(page) => setParams({ ...params, page })}
      />
    </div>
  );
}
```

핵심은 **필터링**이다. 관리자가 가장 많이 할 동작은 "미사용 쿠폰만 보기"와 "특정 회원의 쿠폰 검색"이다. Select 필터로 사용여부(전체/Y/N)를 걸고, 텍스트 검색으로 회원ID나 쿠폰명을 찾는다.

발급현황 테이블의 컬럼:

```typescript
// coupon/issued/columns.tsx
export const issuedColumns: ColumnDef<IssuedCoupon>[] = [
  { accessorKey: "issuedCouponId", header: "발급ID" },
  { accessorKey: "memberId", header: "회원ID" },
  { accessorKey: "couponName", header: "쿠폰명" },
  { accessorKey: "couponCode", header: "쿠폰코드" },
  { accessorKey: "hospitalName", header: "제휴병원" },
  {
    accessorKey: "usedYn",
    header: "사용여부",
    cell: ({ row }) => (
      <Badge variant={row.original.usedYn ? "success" : "secondary"}>
        {row.original.usedYn ? "사용완료" : "미사용"}
      </Badge>
    ),
  },
  { accessorKey: "issuedAt", header: "발급일", cell: formatDateCell },
  { accessorKey: "expireAt", header: "유효기한", cell: formatDateCell },
  {
    accessorKey: "usedAt",
    header: "사용일",
    cell: ({ row }) => row.original.usedAt ? formatDate(row.original.usedAt) : "-",
  },
];
```

사용여부를 `Badge` 컴포넌트로 시각적으로 구분한 건 작은 디테일이지만 가독성에 큰 차이를 만든다. 녹색 "사용완료"와 회색 "미사용"이 한눈에 들어온다.

## 생성/수정된 파일 전체 목록

정리하면 이번 작업에서 만들고 수정한 파일은 이렇다.

| 구분 | 파일 | 작업 |
|------|------|------|
| DB | `ALTER TABLE coupon_list` | 컬럼 2개 추가 |
| Prisma | `schema.prisma` | 필드 2개 추가 |
| 메뉴 | `config/menus.ts` | 수정 |
| 타입 | `coupon/types.ts` | 신규 |
| 훅 | `coupon/hooks/useCouponList.ts` | 신규 |
| 훅 | `coupon/hooks/useIssuedCoupon.ts` | 신규 |
| 쿠폰 목록 | `coupon/list/page.tsx` | 신규 |
| 쿠폰 목록 컬럼 | `coupon/list/columns.tsx` | 신규 |
| 쿠폰 등록 | `coupon/list/add/page.tsx` | 신규 |
| 쿠폰 수정 | `coupon/list/[id]/mod/page.tsx` | 신규 |
| 발급현황 | `coupon/issued/page.tsx` | 신규 |
| 발급현황 컬럼 | `coupon/issued/columns.tsx` | 신규 |
| API | `POST /v2/coupons` | 신규 |
| API | `PUT /v2/coupons/{id}` | 신규 |
| API | `DELETE /v2/coupons/{id}` | 신규 |
| API | `GET /v2/coupons` | 수정 (게시기간 필터 추가) |

신규 파일 10개, 수정 파일 3개, API 4개(신규 3 + 수정 1). 하나의 기능인데 이 정도 분량이 나온다.

## 시리즈를 돌아보며

8~9편에 걸쳐 쿠폰 시스템을 레거시 앱에 심는 과정을 풀어봤다.

**8편**: DB 스키마 설계, 게시기간 개념 도입, 누락된 CRUD API 3개 구현
**9편**: 어드민 프론트엔드 — 등록/수정/삭제 페이지, 발급현황 모니터링

이번 작업에서 가장 크게 느낀 건 **기존 패턴이 있으면 새 기능 추가가 빠르다**는 것이다. React Query 훅 패턴, DataTable 컴포넌트, FilterBar, Tiptap 에디터, 이미지 업로더 — 이전 작업에서 만들어둔 것들을 그대로 가져다 쓰니 화면 하나를 만드는 데 걸리는 시간이 절반 이하로 줄었다.

반대로 말하면, 1~7편에서 레거시를 뜯어고치면서 다진 기반이 이제 빛을 발하기 시작한 거다. 초반에 패턴을 잡느라 시간이 걸렸지만, 한번 잡아놓으니 그 위에 새 기능을 올리는 건 훨씬 수월하다. 레거시 리팩토링의 진짜 가치는 "정리된 코드"가 아니라 "새 기능을 빠르게 만들 수 있는 기반"이었다.

**기획자가 직접 백엔드 API를 만들고, 어드민 프론트까지 올리고, DB 스키마를 설계한다** — 1년 전의 나한테 이 말을 하면 절대 안 믿었을 거다. 근데 바이브 코딩과 패턴화된 구조가 만나니 가능해졌다. 도메인을 아는 사람이 직접 코드를 만지면, 기획서를 누군가에게 전달하고 기다리는 시간이 사라진다. 생각한 걸 바로 만든다. **이 속도감이 스타트업에서는 결정적이다.**
