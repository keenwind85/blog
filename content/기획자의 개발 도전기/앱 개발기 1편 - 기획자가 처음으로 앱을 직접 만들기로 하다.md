# 기획자가 처음으로 앱을 직접 만들기로 하다

## 기획/비즈니스만 10년 넘게, 코드는 한 줄도 못 쓰던 사람

나는 모바일/IT 업계에서 10년 넘게 기획자로 살았다. 사업기획, 서비스기획, PM/PO — 직무명은 바뀌었지만 본질은 같았다. 경영진의 요구사항을 정리하고, 비즈니스 정책과 주요 화면을 설계하고, 개발, 디자이너들과 소통한다. "이렇게 만들어주세요"라고 쓰고, "잘되고 있나요?"하고 점검하는게 나의 일이었다.

때문에 시장을 어떻게 봐야하는지 매출, 비용설계는 어떻게 가져가야 할지, 고객관점에서 웹이 좋을지 모바일이 좋을지, 어떤 기술을 도입하는게 좋을지, API를 쓰는게 좋을지 자체 개발하는게 나을지, 화면 구성은 어떻게 가져가는게 좋을지 등등... 비즈니스와 도메인, 기획, 전략 등에 대한 이해도는 높았으나 한가지 갈증이 있었다면 내가 기획한 서비스가 구체적으로 어떻게 돌아가는지, 코드 레벨에서는 이해하지 못한다는 사실이 답답했다. 개발자에게 "이 API 응답이 느린 것 같은데요"라고 말해도 "서버 사양 문제입니다"라는 대답이 오면, 거기서 대화가 끝났다. 내가 확인할 방법이 없었으니까.

그런데 2025년부터 AI모델이 급속도로 발전하면서 모델이 만들어주는 코드가 꽤나 그럴싸해지기 시작했다. 파이썬 공부한다고 "Hello World"로 호기롭게 시작하다가 본업에 쫓기다보니 코딩공부를 맨날 미뤄두기만 했었는데 어느순간 AI에게 "이런 걸 만들어줘"라고 말하면 코드를 작성해주는 방식. 처음엔 반신반의했지만, 한번 해보기로 했다. 기획서만 쓰던 사람이 진짜 앱을 만들어서 타겟 고객을 대상으로 배포까지 해보는 거다.

## 뭘 만들까 — A.K.A [순시미네](https://play.google.com/store/apps/details?id=com.ssm.care_facility_map)의 탄생

우리 회사는 돌봄 매칭 서비스를 운영하고 있었다. 보호자와 케어매니저를 연결해주는 플랫폼이다. 그런데 서비스를 운영하면서 반복적으로 받는 질문이 있었다.

> "우리 어머니 요양등급 나왔는데, 좋은 요양원 어디 있나요?"

매칭 서비스와는 별개로, **요양시설을 찾고 비교하는 도구**에 대한 수요가 컸다. 전국에 요양원, 주야간보호센터, 방문요양 기관이 수만 개인데, 이걸 한눈에 보고 비교할 수 있는 앱이 없었다.

그래서 결심했다. **"순시미네"** — 순시미의 요양시설 지도 앱을 직접 만들어보자. 
대표님에게 서브 프로젝트를 제안했다. 기존에 만들어진 서비스의 레거시 코드를 건드리는 게 아니라, 완전히 새로운 앱을 AI코딩을 활용해 내가 처음부터 만드는 거다.

## 왜 Flutter + Supabase인가

기술 스택을 고르는 것부터 고민이었다. 기존 회사의 레거시 서비스는 React Native + Next.js API Routes + Java+SprringBoots 조합이었지만, 이번엔 다른 걸 해보고 싶었다.

### Flutter를 선택한 이유

| 고려 사항 | React Native | Flutter |
|-----------|-------------|---------|
| 크로스 플랫폼 | ✅ iOS + Android | ✅ iOS + Android |
| 네이티브 성능 | 🟡 브릿지 통신 | ✅ 직접 네이티브 렌더링 |
| 지도 SDK | 🟡 WebView 경유 가능 | ✅ 네이버 지도 Flutter 플러그인 |
| 새로 배울 동기 | ❌ 기존과 동일 | ✅ 새로운 도전 |

결정적이었던 건 **네이버 지도**였다. 한국 서비스에서 지도는 네이버 지도가 사실상 표준이다. Flutter에는 `flutter_naver_map` 플러그인이 잘 되어 있었고, 네이티브 렌더링이라 성능도 좋았다.

솔직히 말하면, "기존 기술과 다른 걸 해보고 싶다"는 동기도 컸다. 어차피 바이브 코딩으로 할 건데, 새로운 걸 배우는 기회로 삼고 싶었다.

### Supabase를 선택한 이유

백엔드를 어떻게 할지가 더 큰 고민이었다. 선택지는 세 가지였다.

1. **직접 서버 구축** (Node.js + Express) — 관리 부담이 너무 크다
2. **Firebase** — NoSQL(Firestore)은 관계형 데이터에 약하다
3. **Supabase** — PostgreSQL 기반이라 SQL을 쓸 수 있고, 인증/스토리지/Edge Function까지 한 방에 해결

Supabase를 선택한 결정적 이유는 **PostGIS**였다. 요양시설 지도 앱인 만큼 위치 기반 검색이 핵심인데, PostgreSQL의 PostGIS 확장을 쓰면 SQL 한 줄로 "내 위치에서 반경 5km 안의 시설"을 검색할 수 있었다. 이걸 Firestore로 하려면 GeoHash니 뭐니 복잡해진다.

```
기술 스택 결정:
  프론트: Flutter 3.7 (iOS + Android)
  백엔드: Supabase (PostgreSQL + PostGIS)
  인증:   소셜 로그인 (카카오/네이버/애플)
  지도:   네이버 지도 SDK
  AI:    Google Gemini API
  기타:   Firebase (FCM, Remote Config, Analytics)
```

## 디자이너 한 명과의 여정

개발은 나 혼자지만, 디자이너 한 명이 함께했다. UI/UX디자인과 에셋 제작을 맡아주었다. 사실 이 조합이 꽤 잘 맞았다.

기획자인 내가 화면 흐름과, 주요 로직, 기능 명세를 잡으면, 디자이너가 비주얼을 입히고, 내가 바이브 코딩으로 구현한다. 기존에는 기획 → 디자인 → 개발이라는 세 단계가 있었는데, 기획과 개발이 한 사람으로 합쳐지니까 **의사결정이 빨라졌다**.

"이 버튼 위치 어디가 좋을까요?"라고 물으면, 내가 직접 코드에서 바꿔보고 "이쪽이 나은 것 같아요"라고 바로 확인할 수 있었다. 예전에는 이게 JIRA 티켓 한 장이었다.

## 바이브 코딩, 이런 거구나

처음 AI에게 코드를 시켜본 순간이 아직도 생생하다. **(이 당시에는 ClaudeCode도 출시 전이라 무려 커서+GPT 4.x ver의 모델로 바이브코딩을 했다)** Flutter 프로젝트를 생성하고, "로그인 화면을 만들어줘"라고 했더니 진짜 코드가 나왔다. 그런데 그걸 앱에 넣으면 에러가 나고, 에러를 다시 AI에게 보여주면 고쳐주고, 고치면 또 다른 에러가 나고...

처음 한 달은 이 사이클의 연속이었다. 그런데 신기한 건, **이 과정에서 내가 배우고 있었다**는 거다. AI가 짜준 코드를 읽으면서 "아, Dart에서는 이렇게 하는구나", "StatefulWidget이 이런 뜻이구나"를 하나씩 알아갔다.

물론 한계도 있었다. AI는 개별 기능은 잘 만들어주지만, **전체 구조를 잡아주지는 못했다**. 화면이 5개를 넘어가면서부터 "이 서비스 로직을 어디에 두지?", "데이터를 어떻게 공유하지?"라는 질문이 나왔고, 이건 내가 직접 결정해야 했다.

그래서 나온 구조가 **싱글톤 서비스 패턴**이다. `UserService`, `LocationService`, `SupabaseService` 같은 서비스 클래스를 만들어서, 앱 전체에서 하나의 인스턴스를 공유하는 방식. 정식 상태관리 라이브러리(Provider, Riverpod 같은 것)를 쓰지 않은 건, 그때 내가 그런 게 있는지도 몰랐기 때문이다.

돌이켜보면 이게 오히려 장점이었을 수도 있다. 복잡한 상태관리 프레임워크를 끼워넣는 대신, 내가 이해할 수 있는 수준의 단순한 구조로 갔다. 코드가 예쁘지는 않지만, 동작한다. 그리고 내가 전부 이해한다.

## 첫 걸음을 떼며

프로젝트 초기 설정에서 내린 결정들을 정리하면:

| 결정 사항 | 선택 | 이유 |
|-----------|------|------|
| UI 반응형 | flutter_screenutil (375x812 기준) | 다양한 해상도 대응 |
| 폰트 | Pretendard | 한국어 가독성 최고 |
| 앱 진입 | Lottie 스플래시 → 로그인 체크 | 로딩 시간 자연스럽게 가리기 |
| 하단 네비게이션 | 4탭 (지도, AI추천, 커뮤니티, 마이페이지) | 핵심 기능 직관적 접근 |
| 환경 분리 | .env / .env.prod | 개발/운영 환경 분리 |

이렇게 기본 뼈대를 잡는 데만 2주가 걸렸다. 기획서로는 "앱 기본 구조 설정"이라고 한 줄이면 될 것을, 실제로는 Flutter SDK 설치, Supabase 프로젝트 생성, 네이버 지도 API 키 발급, 소셜 로그인 앱 등록, Firebase 프로젝트 연동... 끝이 없었다.

근데 이게 재미있었다. 10년 동안 "설정 좀 해주세요"라고 부탁만 하던 것들을 직접 하나씩 해나가는 경험. 화면에 "Hello World" 대신 우리 로고가 뜨는 순간의 뿌듯함은 어떤 기획서 발표보다 강렬했다.

## 이번 편을 마치며
