# AI 도입 후, 기획자와 개발자의 역할은 어떻게 바뀔까

## 요즘 들어 자꾸 드는 생각

나는 원래 기획자였다. 사업기획, 서비스기획, PM/PO로 10년 넘게 일했다. 그러다 2025년부터 바이브 코딩을 본격적으로 붙잡았고, 지금은 앱을 직접 만들고, 백엔드를 리팩토링하고, 배포까지 한다.

처음에는 단순히 "AI 덕분에 기획자도 개발을 할 수 있게 됐다" 정도로 생각했다. 그런데 실제로 몇 개 프로젝트를 끝까지 밀어보니, 체감은 그보다 훨씬 컸다.

바뀐 건 생산성만이 아니었다. **역할의 경계 자체가 흔들리고 있었다.**

예전에는 이랬다.

```
기획자
  → 요구사항 정리
  → 정책 문서 작성
  → 화면 플로우 정리

개발자
  → 문서를 받아 구현
  → 배포
  → 장애 대응
```

지금은 점점 이렇게 바뀌고 있다.

```
문제를 정의하는 사람
  → 기능과 정책을 설계
  → AI와 함께 직접 구현
  → 배포 후 데이터와 피드백으로 검증

깊은 기술 문제를 해결하는 사람
  → 성능, 보안, 인프라, 데이터 정합성, 대규모 트래픽 대응
  → 시스템이 커질 때 구조를 다시 잡음
```

이 변화를 보면서, 앞으로 시장이 어떻게 나뉠지 조금씩 보이기 시작했다.

## 시장의 변곡점은 "코드를 짜주는 것"이 아니었다

AI가 코드를 작성해주는 건 이미 익숙한 이야기가 됐다. 그런데 실무에서 더 크게 느껴지는 변화는 다른 데 있다.

### 1. 구현 난이도보다 구현 진입 비용이 먼저 내려갔다

예전에는 아이디어가 있어도 바로 만들기 어려웠다. 화면 하나 붙이려면 프론트 개발자, 백엔드 개발자, 기획자, 디자이너가 순서대로 움직여야 했다.

지금은 다르다.

기획자가 플로우를 짜고, AI에게 프롬프트를 잘 넣고, 에러를 읽고, 구조를 조금만 이해해도 **작동하는 프로토타입**까지는 훨씬 빨리 간다.

즉, "만들 수 있느냐 없느냐"의 문제보다 "누가 더 빨리 검증하느냐"의 문제로 바뀌고 있다.

### 2. 주니어에게 주어지던 반복 구현 업무가 가장 먼저 줄고 있다

간단한 CRUD 화면, 단순 API 연결, 반복되는 어드민 페이지, 기본적인 폼 처리 같은 일은 AI가 제일 잘 먹는 구간이다.

예전에는 이런 업무가 주니어 개발자의 훈련장이었다. 지금은 그 영역이 가장 먼저 자동화되고 있다.

이건 꽤 큰 문제다. 단순히 "주니어 일자리가 줄어든다"는 차원을 넘는다. **미래의 시니어를 키우는 파이프라인 자체가 약해질 수 있다.**

### 3. 반대로 깊은 전문성의 가치는 더 커지고 있다

AI가 평균적인 구현을 빠르게 만들수록, 진짜 어려운 문제를 푸는 사람은 더 귀해진다.

예를 들면 이런 영역이다.

- 대규모 인프라와 트래픽
- 인증/권한/보안
- 데이터 정합성과 마이그레이션
- 결제, 검색, 추천
- 모바일 성능 최적화
- 실시간 동기화
- 장애 대응과 관측성

이건 단순히 코드를 많이 아는 문제가 아니다. **실패 비용이 큰 영역을 책임질 수 있느냐**의 문제다.

## 내가 느끼는 기획자의 역할 변화

예전의 기획자는 많은 경우 "정리하는 사람"이었다.

- 요구사항을 모은다
- 기능 명세를 쓴다
- 화면 흐름을 정리한다
- 개발팀과 디자인팀에 전달한다

이 역할이 없어지진 않겠지만, 이 역할만으로는 점점 약해질 가능성이 크다.

왜냐하면 이제는 AI라는 번역기가 생겼기 때문이다. 코드를 모르는 기획자도, 예전보다 훨씬 구체적인 결과물을 직접 만들 수 있다.

그래서 앞으로 기획자에게 더 중요해질 건 이런 것들이다.

- 문제를 정확하게 정의하는 능력
- 정책을 예외 케이스까지 분해하는 능력
- 사용자 흐름을 기능/데이터 구조로 연결하는 능력
- AI에게 모호하지 않은 프롬프트를 주는 능력
- 직접 만든 결과물을 검증하고 수정하는 능력

내가 앱을 직접 만들면서 가장 크게 느낀 것도 이 지점이었다. 기획자의 강점은 여전히 유효했다. 사용자가 어디서 막히는지, 어떤 정보가 먼저 보여야 하는지, 어떤 정책이 운영 리스크를 줄이는지 판단하는 감각은 AI가 대신해주지 않았다.

하지만 동시에 한계도 분명했다. 기획 감각만으로는 부족했다. DB 구조, 인증 방식, 배포 환경, 에러 처리, 토큰 수명 같은 문제는 직접 부딪혀 보기 전까지는 절대 체감할 수 없었다.

결국 앞으로의 기획자는 **문서를 잘 쓰는 사람**보다 **문제를 설계하고 직접 검증하는 사람** 쪽으로 이동할 가능성이 크다.

## 내가 느끼는 개발자의 역할 변화

개발자의 역할도 마찬가지로 바뀌고 있다.

예전에는 "주어진 명세를 얼마나 안정적으로 구현하느냐"가 중요한 구간이 있었다. 물론 지금도 중요하다. 다만 AI가 들어오면서, 구현 그 자체만으로 차별화하기는 어려워지고 있다.

앞으로 개발자에게 더 중요해지는 건 이런 능력이라고 본다.

- 어떤 구조를 선택해야 나중에 안 무너지는지 판단하는 능력
- AI가 짜준 코드를 검증하고 위험을 찾아내는 능력
- 기술적 선택이 제품과 운영에 어떤 영향을 주는지 설명하는 능력
- 기획 의도와 비즈니스 맥락을 이해하고 역으로 제안하는 능력

즉, 개발자도 더 이상 "코드 생산자"에만 머물기 어렵다. 좋은 개발자는 점점 더 **구현자 + 설계자 + 검증자 + 협업 파트너**에 가까워진다.

## 그래서 앞으로 어떤 역할이 커질까

나는 크게 두 축으로 나뉠 거라고 본다.

## 1. 프로덕트 엔지니어

내가 붙여본 이름이다. 아직 정식 직군명으로 완전히 굳은 건 아니지만, 이 표현이 가장 가깝다.

### 프로덕트 엔지니어란

**문제 정의부터 기능 설계, AI 활용 구현, 배포, 운영 검증까지 넓게 책임지는 사람**이다.

기획자와 개발자 사이의 왕복 비용을 줄이고, 작은 팀에서 0→1과 1→10을 빠르게 만드는 역할이다.

### 이 사람들은 무슨 일을 하나

- 사용자 문제를 기능으로 번역한다
- 정책과 예외 케이스를 정리한다
- 화면, API, DB 구조를 함께 설계한다
- AI 도구를 활용해 프론트/백엔드 코드를 직접 생산한다
- 배포 후 지표와 피드백을 보고 바로 수정한다
- 디자이너, 운영, 사업, 개발 사이의 맥락 손실을 줄인다

예전처럼 "기획은 여기까지, 개발은 저기서부터"라고 선을 긋기보다, **프로덕트를 끝까지 굴리는 사람**에 가깝다.

### 수요처는 어디일까

- 스타트업 전반
- 신규 서비스 TF
- 사내 신사업 조직
- PMF를 찾는 초기 제품팀
- 인원이 적고 속도가 중요한 조직

작은 팀일수록 강하다. 한 사람이 넓게 커버할수록 조직의 의사결정 비용이 낮아지기 때문이다.

## 2. 버티컬 엔지니어

이건 내가 임시로 붙인 이름인데, 개인적으로는 꽤 직관적이라고 본다.

### 버티컬 엔지니어란

**특정 기술 카테고리의 복잡도와 실패 비용을 깊게 책임지는 사람**이다.

프로덕트 엔지니어가 넓게 책임진다면, 버티컬 엔지니어는 깊게 해결한다.

### 이 사람들은 무슨 일을 하나

- 고트래픽 환경에서 병목을 진단하고 구조를 바꾼다
- 보안, 권한, 결제, 데이터 정합성 문제를 설계한다
- 장애 대응 체계와 관측성을 만든다
- 공통 플랫폼과 기술 표준을 정리한다
- 시스템이 커졌을 때 기술 부채를 걷어낸다

이 영역은 AI가 대신 짜준 코드만으로는 버티기 어렵다. 도메인 지식, 실패 사례, 트레이드오프 판단이 훨씬 중요하다.

### 수요처는 어디일까

- 스케일업 단계 기업
- 금융, 헬스케어, 커머스, SaaS, 플랫폼, 게임
- 보안과 규제가 중요한 조직
- 많은 사용자를 안정적으로 처리해야 하는 서비스
- 공통 인프라나 코어 시스템을 책임지는 팀

처음부터 버티컬 엔지니어가 많이 필요한 조직도 있지만, 많은 회사는 보통 **제품이 성장하면서 뒤늦게 절실해지는 역할**에 가깝다.

## 그러면 팀 구조는 어떻게 바뀔까

내가 보기에는 점점 이런 그림이 많아질 것 같다.

```
[초기]
프로덕트 엔지니어 중심
  → 빨리 만든다
  → 빨리 검증한다
  → 고객 반응을 본다

[성장]
프로덕트 엔지니어 + 버티컬 엔지니어
  → 병목이 생긴다
  → 보안/성능/정합성 이슈가 드러난다
  → 깊은 전문성이 붙는다

[성숙]
제품 스쿼드 + 전문 조직 공존
  → 넓게 책임지는 사람과
  → 깊게 해결하는 사람이 함께 간다
```

즉, 앞으로는 "기획자냐 개발자냐"보다 **넓게 책임지는 사람이냐, 깊게 해결하는 사람이냐**가 더 중요한 구분이 될 가능성이 크다.

## 그럼 지금의 주니어 개발자와 기획자는 어떻게 해야 할까

이 부분이 가장 중요하다.

AI 시대에 주니어가 사라진다고 말하는 건 쉽다. 하지만 진짜 중요한 건, **그럼 이제 어떻게 커리어를 쌓아야 하느냐**다.

내 생각은 이렇다.

## 1. 주니어 개발자: "코드 작성자"에서 빨리 벗어나야 한다

예전에는 이런 성장 경로가 있었다.

```
작은 티켓 처리
  → 반복 구현 경험 축적
  → 점점 복잡한 기능 담당
  → 시니어로 성장
```

이 경로가 예전보다 약해지고 있다. 그러면 주니어 개발자는 처음부터 이렇게 가야 한다.

### 필요한 방향

- AI가 만든 코드의 오류를 읽고 고칠 수 있어야 한다
- 기능 하나를 프론트/백엔드/DB/배포까지 연결해봐야 한다
- 로그, 모니터링, 에러 원인을 직접 보는 습관이 있어야 한다
- "왜 이 구조를 택했는지"를 설명할 수 있어야 한다
- 제품 맥락을 이해하고 기능의 목적을 알아야 한다

즉, 단순히 코드를 빠르게 짜는 사람보다 **작동하는 시스템을 끝까지 책임져 본 사람**이 강해질 것이다.

## 2. 주니어 기획자: 문서 작성자에 머물면 위험하다

기획자도 마찬가지다.

요구사항 정리, 화면 정의서, 정책 표 작성만으로 커리어를 쌓는 방식은 점점 약해질 가능성이 크다. AI가 문서 초안도 만들고, 화면 구조도 정리하고, 심지어 시안 수준의 프로토타입도 만들어주기 때문이다.

### 필요한 방향

- 기능을 사용자 흐름뿐 아니라 데이터 흐름으로도 이해해야 한다
- API, DB, 인증, 상태 관리가 어떤 식으로 제품에 연결되는지 알아야 한다
- Figma나 문서에서 멈추지 말고 직접 프로토타입까지 만들어봐야 한다
- AI에게 좋은 결과를 뽑는 프롬프트 작성 능력을 길러야 한다
- 운영 관점에서 예외 케이스와 정책 충돌을 설계할 수 있어야 한다

앞으로의 기획자는 "정리 잘하는 사람"보다 **제품을 더 구체적으로 설계하고 실행까지 연결하는 사람**이 유리할 가능성이 크다.

## 3. 결국 주니어는 둘 중 하나를 빨리 선택해야 한다

나는 앞으로 주니어가 이렇게 나뉠 거라고 본다.

### 프로덕트 엔지니어 지향

- 작은 제품을 직접 끝까지 만들어본다
- 기획, 화면, API, DB, 배포를 넓게 경험한다
- 속도와 맥락 이해가 강점이 된다

### 버티컬 엔지니어 지향

- 특정 기술 축을 깊게 판다
- 네트워크, DB, 보안, 인프라, 성능 등 기반기를 더 탄탄히 한다
- 난도가 높은 문제를 푸는 힘이 강점이 된다

둘 다 괜찮다. 중요한 건 **예전처럼 막연하게 "일단 주니어 개발자" 혹은 "일단 주니어 기획자"로 버티기 어려워질 수 있다**는 점이다.

## 내가 생각하는 현실적인 대응 방법

조금 더 현실적으로 쓰면 이렇다.

### 지금의 주니어 개발자에게

1. AI 툴은 무조건 익혀야 한다  
2. 작은 서비스 하나를 혼자 끝까지 만들어봐야 한다  
3. 동시에 한 축은 깊게 파야 한다  
4. "구현했다"보다 "운영해봤다"가 중요해진다  

### 지금의 주니어 기획자에게

1. 기능 명세만 쓰지 말고 직접 프로토타입을 만들어봐야 한다  
2. 개발자와 대화할 때 API, DB, 인증 구조를 이해해야 한다  
3. 정책 설계에서 예외 케이스를 더 집요하게 봐야 한다  
4. AI를 문서 보조 도구가 아니라 실행 도구로 써야 한다  

## 결국 남는 사람은 누구일까

내 생각에 앞으로 살아남는 사람은 크게 두 부류다.

- **넓게 책임지는 사람**
- **깊게 해결하는 사람**

반대로 가장 애매해질 수 있는 사람은,

- 문서만 쓰는 기획자
- 시키는 것만 구현하는 개발자

물론 현실은 이렇게 깔끔하게 나뉘지 않는다. 회사마다 다르고, 산업마다 다르고, 사람마다 속도도 다르다. 하지만 큰 흐름은 분명히 보인다.

AI는 기획자도 개발자도 바로 대체하지 않을 것이다. 대신 **양쪽 모두에게 더 높은 수준의 역할 이동을 요구할 가능성**이 크다.

나 역시 그 변화 한가운데에 있다고 느낀다. 기획만 하던 시절에는 몰랐던 구현의 무게를 배웠고, 개발을 하면서는 기획의 중요성을 다시 확인했다. 결국 중요한 건 직함이 아니라, **문제를 끝까지 책임질 수 있느냐**인 것 같다.

앞으로 5년, 10년 뒤 직군 이름은 바뀔 수 있다. 프로덕트 엔지니어라는 말이 정착할 수도 있고, 다른 이름이 붙을 수도 있다. 하지만 방향 하나는 꽤 분명해 보인다.

> **기획과 개발의 경계는 흐려지고, 넓게 책임지는 사람과 깊게 해결하는 사람이 더 중요해진다.**

이 변화 속에서 나는 계속 양쪽 언어를 다루는 사람이 되고 싶다. 문제를 정의할 줄 알고, 필요하면 직접 만들 줄 알고, 어디서부터 전문성을 붙여야 하는지도 아는 사람. 아마 앞으로 시장이 가장 많이 찾는 사람도 그런 쪽에 가까울 거라고 생각한다.
