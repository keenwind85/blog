# 블로그 소재 발굴 (blog-surver)

업무 문서 폴더(`/Users/wook/Desktop/SSM_PLAN/`)를 스캔하고, 인터넷 트렌드 검색을 결합하여 블로그에 올릴 만한 소재 후보를 선별해서 제안합니다. 사용자가 후보를 선택하면 `blog-write` 스킬로 이어서 글을 작성합니다.

## 왜 이 스킬이 필요한가

블로그 글감은 보통 업무 중에 쌓인다. 하지만 막상 "뭘 쓸까?" 하면 떠오르지 않는다. 이 스킬은 이미 쌓여 있는 업무 기록을 훑어보고, 거기서 블로그에 올릴 만한 이야기를 뽑아주는 역할을 한다. 추가로 요즘 기술 트렌드와 맞물리는 소재를 찾아서, 독자에게도 시의성 있는 글이 되도록 돕는다.

## 실행 절차

### 1단계: 기존 블로그 글 목록 수집 (중복 제거용)

이미 블로그에 올린 글의 주제를 먼저 파악해야 한다. 같은 이야기를 다시 쓰면 의미가 없기 때문이다.

1. `/Users/wook/Desktop/BLOG/content/index.md`를 읽는다
2. 각 시리즈와 개별 글 제목을 목록으로 정리한다
3. `/Users/wook/Desktop/BLOG/content/` 하위 디렉토리의 모든 .md 파일명도 수집한다
4. 이 목록을 **"이미 다룬 주제 목록"**으로 저장해둔다

이 목록은 이후 모든 소재 후보를 필터링하는 기준이 된다.

### 2단계: 업무 문서 스캔 및 소재 추출

`/Users/wook/Desktop/SSM_PLAN/` 폴더 전체를 탐색한다.

#### 스캔 범위
- 루트 레벨의 .md 파일
- 하위 디렉토리의 .md 파일 (재귀 탐색)
- .pptx, .html 등 비 마크다운 파일은 제목/파일명만 참고

#### 스캔 제외 대상
- `.obsidian/`, `.claude/`, `.vendor/` 등 시스템 디렉토리
- `채용/`, `첨부파일/` 등 개인정보가 포함된 디렉토리 (이력서, 채용 분석 등)
- 단순 설정 파일, 로그 파일 (firebase-debug.log 등)

#### 각 문서에서 추출할 정보
문서를 읽고 다음을 파악한다:
- **핵심 주제**: 이 문서가 다루는 기술/업무의 핵심이 무엇인가
- **경험의 깊이**: 삽질 기록, 의사결정 과정, 구현 상세가 있는가 (이야기가 될 만한가)
- **독자 관심도**: 다른 개발자/기획자도 공감하거나 배울 수 있는 내용인가
- **블로그 적합도**: 회사 기밀이나 민감 정보 없이 공개 가능한가

#### 소재 후보 판단 기준
좋은 블로그 소재는 다음 중 2개 이상을 충족한다:
- 직접 해본 경험이 담겨 있다 (삽질, 실패, 해결 과정)
- 기술적 구현과 비즈니스 의사결정이 교차한다
- 다른 기획자나 개발자에게 실질적 도움이 된다
- 시리즈로 확장 가능하다
- 현재 기술 트렌드와 맞물린다

#### 중복 제거
- 1단계에서 수집한 "이미 다룬 주제 목록"과 대조한다
- 이미 블로그에 올린 주제(백엔드 리팩토링, 쿠폰 시스템, 푸시 알림, FCM, 카카오 알림톡, 맞고 게임, 앱 개발기, 햄릿 관리, AI 역할 변화, Anthropic Economic Index 등)는 제외한다
- 단, 기존 주제의 **후속편/심화편**은 허용한다 (예: "백엔드 도전기 11편"으로 이어질 수 있는 새 에피소드)

### 3단계: 인터넷 트렌드 검색

웹 검색 도구를 사용하여 최신 기술 트렌드를 조사한다.

#### 검색 키워드 (한국어 + 영어 혼합)
- 기획 관련: "PM PO 역할 변화 2026", "프로덕트 매니저 트렌드", "바이브코딩"
- 개발 관련: "개발자 트렌드 2026", "풀스택 개발", "레거시 마이그레이션"
- AI 관련: "AI agent 개발", "Claude Code", "AI coding assistant 트렌드", "vibe coding"
- 테크 관련: "스타트업 기술 스택 2026", "모바일 앱 개발 트렌드"

#### 트렌드 정보 수집
- 기술 블로그, 뉴스에서 현재 화제가 되는 주제 파악
- 어떤 키워드가 주목받고 있는지 확인
- 2단계에서 추출한 소재 후보와 연결 가능한 트렌드 식별

### 4단계: 소재 후보 종합 및 순위 매기기

2단계(업무 문서)와 3단계(트렌드)를 결합하여 최종 소재 후보를 만든다.

#### 후보 구성 형식
각 후보에 대해 다음을 정리한다:

```
## 후보 N: [제안 타이틀]

- **카테고리**: [기획자의 개발 도전기] / [AI Paradox] / [신규 카테고리 제안]
- **소스 문서**: 어떤 업무 문서에서 발굴했는지
- **핵심 스토리**: 2~3줄로 어떤 이야기를 풀 수 있는지
- **트렌드 연결**: 현재 어떤 기술 트렌드와 맞물리는지
- **예상 분량**: 1편 / 2편 시리즈 등
- **추천 이유**: 왜 이 소재가 좋은지 한 줄 요약
```

#### 순위 기준 (높은 순)
1. 업무 문서의 경험 깊이 + 트렌드 연결성이 모두 높은 소재
2. 경험은 깊지만 트렌드 연결은 약한 소재 (본질적으로 좋은 이야기)
3. 트렌드는 강하지만 업무 경험이 얕은 소재 (시의성 위주)

#### 제시 개수
- 3~5개의 후보를 제시한다
- 너무 많으면 선택 피로, 너무 적으면 선택지 부족

### 5단계: 사용자 선택 및 blog-write 연결

후보를 사용자에게 보여주고, 선택을 기다린다.

#### 선택 프로세스
1. 후보를 제시하며 "어떤 소재로 글을 써볼까요?" 라고 묻는다
2. 사용자가 후보 번호를 선택하거나, 수정 요청을 할 수 있다
3. 사용자가 선택하면 해당 소재의 소스 문서 경로와 핵심 방향을 정리한다

#### blog-write 연결
사용자가 최종 선택하면:
1. 선택된 소재의 소스 문서를 `blog-write` 스킬의 입력으로 전달한다
2. 트렌드 키워드와 방향성도 함께 전달하여, 글 작성 시 반영되도록 한다
3. `blog-write` 스킬의 실행 절차를 따라 글 작성을 시작한다

연결 시 전달할 정보:
```
선택된 소재: [타이틀]
카테고리: [카테고리명]
소스 문서: [파일 경로]
트렌드 키워드: [관련 키워드들]
작성 방향: [핵심 스토리 요약]
```

## 사용 예시
```
/blog-surver
```
→ 업무 문서를 스캔하고, 트렌드를 검색하여, 블로그 소재 후보 3~5개를 제안

## 주의사항
- 채용 관련 문서, 이력서 등 개인정보가 포함된 파일은 소재에서 제외한다
- 회사 기밀(서버 비밀번호, API 키, 내부 전용 비즈니스 데이터 등)이 드러나는 소재는 피한다
- 업무 문서의 "기술적 경험"과 "의사결정 과정"은 블로그에 활용 가능하지만, 구체적인 사업 수치나 고객 데이터는 제외한다
- 트렌드 검색 결과가 업무 문서와 전혀 맞지 않으면 무리하게 연결하지 않는다
