# 개인 블로그 글쓰기

사용자가 제공한 문서(md, 텍스트, URL 등)를 기반으로 블로그 신규 글을 작성합니다.

## 실행 절차

### 1단계: 입력 문서 분석
- 사용자가 제시한 문서($ARGUMENTS)를 읽고 핵심 주제, 분량, 구조를 파악합니다
- 문서의 성격을 판단합니다: 개발 경험기인지, AI/기술 트렌드 분석인지, 기타 새로운 성격인지

### 2단계: 기존 글 톤앤매너 학습
- `/Users/wook/Desktop/BLOG/content/` 하위 기존 글 2~3개를 읽어 톤앤매너를 파악합니다
- 반드시 준수할 글쓰기 스타일:
  - **1인칭 서술**: "나는", "내가" 등 필자 관점으로 서술
  - **경험 기반**: 이론 나열이 아닌 실제 겪은 이야기처럼 풀어냄
  - **솔직한 어투**: "솔직히 말하면", "처음엔 반신반의했지만" 등 속내를 드러내는 표현
  - **기획자 관점**: 기술 자체보다 "왜 이런 결정을 했는지", "비즈니스적으로 어떤 의미인지"에 초점
  - **구어체+문어체 혼합**: 딱딱하지 않으면서도 전문성 유지
  - **실무 중심**: 추상적 설명보다 구체적 사례, 표, 코드블록, 인용구 활용
  - **시리즈 내 연결**: 이전/다음 글 참조 시 `[[]]` 옵시디언 링크 활용

### 3단계: 카테고리 결정
현재 카테고리 2개:
- **[기획자의 개발 도전기]**: 직접 코드를 짜고 서비스를 만드는 개발 경험 기록
- **[AI Paradox]**: AI 시대의 역할 변화, 기술 트렌드, 산업 분석 등 관점/의견 중심 글

판단 기준:
- 코드를 짜거나 서비스를 직접 만든 경험 → `[기획자의 개발 도전기]`
- AI/기술 트렌드 분석, 역할 변화, 산업 관점 → `[AI Paradox]`
- 위 두 카테고리에 맞지 않는 경우 → 글의 맥락에 맞는 **신규 카테고리명을 제안**하고 사용자 확인 후 생성

### 4단계: 분량 판단 및 분할
- 소스 문서의 양과 주제의 깊이에 따라 1편, 2편, 3편... 등으로 분할 여부를 결정합니다
- 분할 기준:
  - 1편당 적정 분량: 한글 기준 약 2,000~4,000자 (읽기 시간 5~10분)
  - 주제가 자연스럽게 나뉘는 단위로 분할 (시간순, 기능순, 단계순 등)
  - 각 편이 독립적으로 읽혀도 의미가 있어야 함
- 분할 계획을 사용자에게 먼저 제안하고 확인 후 작성합니다

### 5단계: 글 작성
프론트매터 형식:
```yaml
---
title: "[카테고리명] 시리즈명 N편 - 부제목"
description: 1~2줄 요약 설명
date: YYYY-MM-DD (작성일)
tags:
  - 태그1
  - 태그2
  - 태그3
---
```

본문 구조:
- `# 큰 제목` (title의 부제목 부분과 동일하거나 유사)
- `## 소제목`으로 섹션 구분
- 표, 코드블록, 인용구(`>`)를 적극 활용
- 각 편의 끝에 다음 편 예고 또는 시리즈 요약

파일 저장 위치:
- `[기획자의 개발 도전기]` → `/Users/wook/Desktop/BLOG/content/기획자의 개발 도전기/`
- `[AI Paradox]` → `/Users/wook/Desktop/BLOG/content/AI Paradox/`
- 신규 카테고리 → `/Users/wook/Desktop/BLOG/content/{카테고리명}/` 디렉토리 생성 후 저장

파일명 규칙: `{시리즈명} {N}편 - {부제목}.md`

### 6단계: index.md 업데이트
- `/Users/wook/Desktop/BLOG/content/index.md`에 새 글의 `[[]]` 링크를 추가합니다
- 기존 시리즈에 추가하는 경우 해당 시리즈 섹션에 링크 추가
- 신규 시리즈/카테고리인 경우 새로운 섹션을 생성하고 간단한 소개 문구 작성

### 7단계: 최종 확인
- 작성된 글을 사용자에게 보여주고 피드백을 받습니다
- 톤앤매너, 분량, 구조에 대한 수정 요청을 반영합니다

## 사용 예시
```
/blog-write @/path/to/source-document.md
/blog-write @/path/to/research-report.pdf
```

## 주의사항
- 소스 문서의 내용을 그대로 옮기지 않고, 필자의 관점과 경험으로 재구성합니다
- 기존 글에서 언급한 배경(10년차 스타트업 PO, 바이브코딩 경험 등)과 일관성을 유지합니다
- 마크다운 문법은 옵시디언 호환 형식을 따릅니다
