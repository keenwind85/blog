# 순시미 파트너스 Cloud PC PoC 및 MVP 실행 계획안

## 1. 문서 목적

본 문서는 실제 제휴센터 1곳을 대상으로 `Cloud PC 기반 크롤링 환경`을 구축하고, `crawler-agent MVP`를 개발한 뒤, 이를 기존 `ssm-app`과 `PARTNER_ADMIN`까지 연결하는 PoC/MVP 실행 순서를 구체적으로 정리한 문서다.

목표는 단순 기술 검증이 아니라, 다음 4가지를 실제 운영 환경에서 end-to-end로 확인하는 것이다.

1. Cloud PC에서 공동인증서 기반 수집이 안정적으로 동작하는가
2. 수집 데이터를 서버 DB에 정규화하여 저장할 수 있는가
3. 보호자 앱에서 계약 인증 후 본인 데이터만 조회할 수 있는가
4. PARTNER_ADMIN에서 승인/반려와 동기화 상태 관리가 가능한가

---

## 2. 이번 PoC/MVP의 범위 정의

## 2.1 PoC 범위

PoC는 아래 질문에 답하기 위한 단계다.

- 센터별 Cloud PC 운영이 가능한가
- 대상 사이트 로그인 및 기본 데이터 수집이 가능한가
- crawler-agent를 설치형으로 운영할 수 있는가
- heartbeat, 동기화 상태 확인, 수동 재실행이 가능한가

즉, PoC는 `기술/운영 가능성 검증`에 집중한다.

## 2.2 MVP 범위

MVP는 아래 흐름이 실제 사용자 가치로 이어지는지 검증한다.

1. 센터 데이터 수집
2. DB 저장 및 정규화
3. 보호자 계약 인증 요청
4. 센터 관리자 승인
5. 보호자 앱에서 계약 데이터 조회

즉, MVP는 `실제 서비스 흐름 검증`에 집중한다.

---

## 3. 이번 1차 검증의 최종 목표

이번 1차 실행의 최종 성공 조건은 아래와 같다.

### 기술 목표

- Cloud PC에서 대상 사이트 로그인 성공
- 계약 관련 핵심 데이터 1종 이상 자동 수집 성공
- 수집 데이터를 서버 DB에 저장 성공
- 보호자 앱과 PARTNER_ADMIN에 연결 성공

### 운영 목표

- 운영자가 원격으로 상태 확인 가능
- 장애 발생 시 로그와 에러 원인 파악 가능
- 최소 3일 이상 반복 동기화 안정 동작

### 제품 목표

- 센터 관리자 1명이 승인/반려를 수행 가능
- 보호자 1명이 계약 인증 요청 후 본인 계약정보를 앱에서 조회 가능

---

## 4. 전체 추진 전략

이번 작업은 한 번에 끝까지 만들기보다, 아래 순서대로 레이어를 하나씩 고정하는 방식으로 가야 한다.

```text
1. Cloud PC 환경 먼저 고정
2. crawler-agent 최소 로그인/수집 루틴 구현
3. 서버 업로드 및 DB 저장 구현
4. 정규화/권한 모델 구현
5. PARTNER_ADMIN 승인 플로우 구현
6. 보호자앱 조회 플로우 연결
7. 반복 수집/관제/운영성 검증
```

핵심은 `UI보다 수집-저장-권한-조회` 흐름을 먼저 뚫는 것이다.

## 5. 단계별 실행 순서

## 5.1 0단계: 사전 정리

### 해야 할 일

- PoC 대상 제휴센터 1곳 확정
- 센터 담당자 1명 지정
- 센터에서 제공 가능한 인증서/로그인 정보 범위 확인
- 대상 사이트의 수집 대상 항목 우선순위 정리
- PoC 성공 기준 합의

### 정리해야 할 항목

- 센터명
- 관리자 담당자
- 공동인증서 사용 방식
- 로그인 절차
- 수집 대상 데이터 목록
- 테스트용 보호자 계정

### 산출물

- PoC 대상 센터 정보 시트
- 수집 대상 정의서
- 성공 기준 체크리스트

---

## 5.2 1단계: Cloud PC 환경 선정 및 세팅

이 단계는 기술보다 중요하다. 여기서 흔들리면 뒤 단계가 전부 밀린다.

### 해야 할 일

- Cloud PC 벤더 선택
- 센터별 전용 persistent Windows 환경 생성
- 운영자 원격 접속 정책 정리
- 브라우저/보안모듈/인증서 설치 가능 여부 확인
- 자동 로그인/재부팅 후 자동 실행 정책 설정

### 권장 세팅 항목

- Windows OS 고정 버전
- Chrome/Edge 고정 버전
- Playwright runtime 설치
- 보안 프로그램 설치
- crawler-agent 설치 디렉토리 표준화
- 로컬 로그 디렉토리 표준화

### 이 단계의 검증 항목

- 운영자가 원격 접속 가능한가
- 대상 사이트 수동 로그인 가능한가
- 공동인증서 수동 로그인 가능한가
- 재부팅 후 환경이 유지되는가

### 성공 기준

- Cloud PC에서 수동 로그인 성공
- 1회 수동 조회 가능
- 재부팅 후 동일 환경 복원 가능

### 실패 시 분기

- 인증서/보안모듈이 Cloud PC에서 불안정하면
  - 센터 전용 물리 미니PC 대안 검토
  - 또는 Cloud PC 이미지/정책 재조정

---

## 5.3 2단계: crawler-agent PoC 버전 개발

이 단계에서는 UI보다도 `로그인 + 조회 + raw 저장 + 업로드`에 집중한다.

### 구현 목표

- agent 등록
- heartbeat
- 수동 실행 트리거
- 로그인
- 계약 관련 기본 데이터 1종 수집
- raw 데이터 업로드

### 최소 기능 범위

- `register`
- `heartbeat`
- `manual sync`
- `auth`
- `crawl basic contract data`
- `upload raw payload`

### 권장 구현 순서

1. agent 실행 및 설정 파일 로드
2. 서버 등록 API 연동
3. heartbeat 전송
4. Playwright 로그인 스크립트 구현
5. 대상 화면 진입
6. HTML/JSON raw 추출
7. SQLite 로컬 저장
8. 서버 업로드

### 이 단계의 산출물

- crawler-agent 최소 실행 앱
- Cloud PC 배포 패키지
- raw upload API
- sync job log 구조

### 성공 기준

- 운영자가 수동 실행 버튼 또는 명령으로 동기화 가능
- raw 데이터가 서버에 올라감
- 실패 시 에러코드와 로그가 남음

---

## 5.4 3단계: 서버 저장 구조 구현

이 단계부터는 크롤러보다 서버 모델링이 중요해진다.

### 구현 목표

- raw 테이블 저장
- normalized 테이블 저장
- partner center와 contract 매핑
- idempotent sync

### 우선 구현 테이블

- `partner_center`
- `partner_agent`
- `partner_sync_job`
- `partner_sync_log`
- `partner_raw_contract`
- `partner_contract`

### 구현 순서

1. Prisma schema 초안 작성
2. raw upsert 로직 작성
3. normalized transform 로직 작성
4. contract unique key 전략 확정
5. 중복 방지 및 변경 감지 처리

### 검증 포인트

- 같은 데이터를 다시 올려도 중복 생성되지 않는가
- 변경 데이터는 갱신되는가
- 센터별 데이터가 섞이지 않는가

### 성공 기준

- 1개 센터의 계약 데이터가 raw/normalized에 모두 저장됨
- 동일 데이터 재수집 시 중복 없음

---

## 5.5 4단계: PARTNER_ADMIN MVP 개발

이 단계에서는 관리자 웹의 화려한 화면보다 승인 흐름과 상태 확인이 핵심이다.

### MVP에서 필요한 화면

- 로그인
- 대시보드
- 링크 요청 목록
- 승인/반려 처리
- 동기화 상태 목록
- 계약 데이터 기본 조회

### 반드시 보여야 하는 정보

- 마지막 성공 동기화 시간
- 최근 실패 여부
- 요청한 보호자 이름
- 연결 대상 어르신/계약 정보
- 승인 상태

### 구현 순서

1. 파트너 관리자 인증/권한 처리
2. 링크 요청 조회 API
3. 승인/반려 API
4. 동기화 상태 API
5. 최소 관리자 화면 구현

### 성공 기준

- 센터 담당자가 로그인 가능
- 보호자 요청을 승인/반려 가능
- 동기화 상태를 확인 가능

---

## 5.6 5단계: 보호자 앱 계약 인증 흐름 구현

이 단계에서 비로소 기존 `ssm-app`과 연결된다.

### MVP에서 필요한 보호자 앱 기능

- 파트너 센터 선택
- 계약 인증 요청
- 승인 상태 확인
- 승인 후 계약 데이터 조회

### 구현 순서

1. 센터 검색/선택 API
2. 계약 인증 요청 API
3. 링크 요청 상태 조회 API
4. 승인된 계약 데이터 조회 API
5. 보호자 앱 최소 화면 연결

### 화면 MVP 기준

- 센터 선택 화면
- 인증 요청 완료 화면
- 승인 대기 상태 화면
- 승인 후 기본 계약 정보 조회 화면

### 성공 기준

- 테스트 보호자가 실제로 요청 가능
- PARTNER_ADMIN에서 승인 가능
- 승인 후 앱에서 본인 계약 정보 조회 가능

---

## 5.7 6단계: end-to-end 통합 검증

이 단계에서는 기능이 아니라 흐름을 본다.

### 검증 시나리오

1. 센터 Cloud PC 정상 동작
2. crawler-agent 동기화 성공
3. 계약 데이터 DB 반영
4. 보호자 인증 요청
5. 관리자 승인
6. 보호자 앱 조회

### 함께 확인해야 할 것

- 권한 범위가 맞는가
- 타 보호자 데이터가 노출되지 않는가
- 계약 변경 후 재수집 시 갱신되는가
- 에러 발생 시 운영자가 즉시 원인 파악 가능한가

### 성공 기준

- 위 시나리오가 운영자 개입 최소로 1회 이상 완주됨
- 실제 센터 담당자 피드백 획득

## 5.8 7단계: 3일~1주 운영 검증

이 단계가 사실상 PoC의 핵심이다.

### 목표

- 반복 동기화 안정성 확인
- 장애 유형 수집
- 운영 방식 보정

### 확인 포인트

- Cloud PC 세션 유지 안정성
- crawler-agent 메모리/프로세스 안정성
- 인증서 로그인 반복 성공률
- sync 실패 패턴
- last successful sync 갱신 여부

### 운영자가 체크할 지표

- 온라인 여부
- heartbeat 간격
- 성공/실패 횟수
- 최근 에러코드
- 수동 재실행 성공률

### 성공 기준

- 3일 이상 반복 동기화 성공
- 치명적 장애 없거나 우회 가능
- 운영 대응 절차 문서화 가능

## 8. 팀 운영 방식 제안

PoC/MVP는 병렬보다도 의존성 순서가 중요하다.

### 추천 작업 순서

1. 인프라/운영: Cloud PC 환경 준비
2. 크롤러 개발: 로그인/수집/업로드
3. 백엔드 개발: 저장/정규화/권한
4. PARTNER_ADMIN 개발: 승인/상태
5. 보호자 앱 연결

### 이유

- Cloud PC가 안 되면 크롤러가 무의미하다
- 크롤러가 raw를 못 올리면 관리자 웹도 무의미하다
- 승인 구조가 없으면 보호자 앱 조회가 위험하다

즉, `크롤러 → DB → 권한 → UI` 순서를 지켜야 한다.

## 10. Go / No-Go 기준

PoC 이후 아래 기준으로 다음 단계 진행 여부를 판단한다.

### Go 조건

- Cloud PC에서 인증서 기반 로그인 안정 동작
- 기본 계약 데이터 수집 성공
- DB 저장 및 앱 조회까지 연결 성공
- 센터 담당자 협조 가능성 확인

### No-Go 또는 재설계 조건

- Cloud PC에서 보안모듈이 반복적으로 불안정
- 수집 대상 사이트 구조가 지나치게 자주 깨짐
- 인증서 custody 이슈가 계약상 해결되지 않음
- 운영자가 장애 원인을 추적할 수 없음

## 11. 이번 실행의 핵심 원칙

이번 1차는 처음부터 완성형 제품을 만들려는 프로젝트가 아니다.

반드시 아래 원칙으로 가야 한다.

- 한 센터, 한 데이터셋, 한 승인 흐름부터 검증한다
- raw 수집보다 권한과 노출 정책을 더 엄격하게 본다
- 자동화 실패를 전제로 fallback을 남긴다
- 운영자가 직접 볼 수 있는 로그와 상태를 먼저 만든다
- UI 완성도보다 end-to-end 연결을 우선한다

## 13. 최종 정리

이번 제휴센터 1곳을 대상으로 한 PoC/MVP는 다음 순서로 진행하는 것이 가장 현실적이다.

1. 센터와 PoC 범위 확정
2. Cloud PC 환경 세팅
3. crawler-agent 최소 기능 구현
4. raw/normalized DB 저장 구현
5. PARTNER_ADMIN 승인 흐름 구현
6. 보호자 앱 조회 흐름 연결
7. 실제 운영 환경에서 반복 검증

핵심은 `Cloud PC + crawler-agent + DB + 권한 + 앱 조회`를 한 번에 연결해 보는 것이다.

이 흐름이 실제로 돌아가면, 그다음부터는 데이터셋 확장, 문서/결제/채팅, 운영 자동화로 확장할 수 있다. 반대로 이 흐름이 안 돌아가면, 스택보다도 운영 구조와 제품 정의를 먼저 다시 봐야 한다.
